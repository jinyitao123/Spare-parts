## 第一章 设计哲学：为什么是 Agent Native

### 1.1 传统路径的死局

传统路径是线性的：先建系统 → 再积累数据 → 最后上 AI。

永祥科技的数据现实让这条路径走进死局：二级库无记录、故障记录纸质、保养计划不含备件清单、采购周期从 2 天到半年不等。如果按传统路径走，MVO 上线后的 12 个月内客户看到的只是一个"电子表格升级版"，感知不到任何 AI 价值。

### 1.2 Agent Native 的翻转

Agent Native 的核心翻转是：**Agent 不是数据的消费者，Agent 是数据飞轮的启动器。**

传统思路——数据好了才上 AI。
Agent Native 思路——Agent 从第一天起参与业务流程，在帮用户完成日常操作的同时，顺便把高质量数据积累起来。

| 维度 | 传统路径 | Agent Native |
|---|---|---|
| 数据采集 | 表单/扫码 → 人填什么就是什么 | Agent 对话中自然获取上下文（故障原因、设备关联） |
| 数据质量 | 依赖人的自觉 | Agent 校验合理性、追问缺失字段 |
| 管控能力 | 规则写在手册里，执行看个人 | Agent 在行动前嵌入判断，异常前置拦截 |
| AI 价值感知 | 12 个月后才有 | Day-1 就有——每一次领用都有 Agent 参与 |
| 数据飞轮 | 不存在 | Agent-1 的日常交互为 Agent-7/8/9 积累训练数据 |

### 1.3 二级库的本质问题

扫码解决的是**记录问题**——从"不记"变成"记了"。但二级库的本质问题不是"没记录"，而是**没管控**。

| 环节 | 扫码解决了吗 | Agent 解决什么 |
|---|---|---|
| 谁拿了什么、几个 | ✅ | — |
| 该不该拿这个？型号对不对？ | ❌ | 需求校验：历史消耗 + 规格匹配 |
| 拿 2 个合理吗？ | ❌ | 消耗合理性判断：历史同类故障用量 |
| 有更便宜的兼容型号吗？ | ❌ | 替代件推荐：规格匹配 + 成本优化 |
| 这台设备本月第 3 次换同一个件了 | ❌ | 异常检测：频次分析 + 根因提示 |
| 领了没用上，什么时候退？ | ❌ | 闭环跟踪：未消耗提醒 + 退库引导 |
| 该用呆滞库存里的件但没人知道 | ❌ | 呆滞引导：领用时主动推荐呆滞件消化 |

**结论：扫码把混乱数字化了，Agent 把混乱变成秩序。**

## 第二章 双通道架构（方案C）

### 2.1 为什么是双通道

不是所有场景都适合同一种交互模式。故障抢修时工程师在机台旁边手忙脚乱，让他先跟 Agent 聊天不现实。保养计划性领用时有充裕时间，Agent 可以做深度引导。

### 2.2 两个通道的定义

**通道 A：Agent 驱动流程（计划性领用）**

适用场景：保养领用、技改领用、项目实施、计划性更换

```
工程师对 Agent 描述需求
  → Agent 理解意图，提取：故障现象 / 保养项目 / 关联设备 / 所需备件
  → Agent 校验合理性（历史消耗、库存状态、替代件、呆滞件）
  → Agent 生成领用指令（物料编码、数量、原因、设备、建议方案）
  → 工程师确认
  → 工程师去货架取件 → 扫码验证（验证拿对了没）
  → Agent 记录闭环，启动后续跟踪
```

关键点：Agent 是主流程，扫码是末端物理校验。领用原因、关联设备等元数据在对话中自然获取，不需要工程师额外填表。

**通道 B：扫码快速通道 + Agent 异步补全（紧急领用）**

适用场景：故障抢修、突发停机

```
工程师直接去库房 → 扫码出库（标记为"紧急领用"）
  → 系统立即记录出库
  → Agent 在 15 分钟内异步发起轻量确认
    "刚才领的 6205 轴承，是 3 号线电机故障换件吗？"
    工程师回复 "是" 或补充信息
  → Agent 补全记录（故障原因、关联设备）
  → 启动后续跟踪
```

关键点：不阻塞紧急操作，但 Agent 及时跟进补全。15 分钟窗口内工程师通常已完成更换，有空余回复。

### 2.3 通道选择逻辑

```
领用请求到达
  │
  ├─ 工程师主动对 Agent 说话 ──→ 通道 A（Agent 驱动）
  │
  └─ 工程师直接扫码 ──→ 通道 B（快速通道）
       │
       └─ Agent 15 分钟内异步补全
```

不需要规则引擎判断走哪条通道——工程师的行为本身就是选择。跟 Agent 说话就走 A，直接扫码就走 B。系统不强制，靠价值引导：走通道 A 能得到更好的建议（替代件、成本优化、异常提醒），时间长了工程师会自然倾向于先问 Agent。

### 2.4 高风险拦截（跨通道）

无论走哪个通道，以下情况触发强制拦截：

| 拦截条件 | 行为 | 原因 |
|---|---|---|
| 备件单价 > 2000 元 | 扫码时弹窗确认 + Agent 提醒 | 高价值备件有退库报废要求 |
| 该备件是安全库存最后一个 | Agent 警告 + 建议同步触发采购 | 安全库存见底 |
| 同一设备本月第 3 次领用同一备件 | Agent 异常提醒 + 建议排查根因 | 可能存在系统性故障 |
| 领用备件与设备不匹配（BOM 校验） | Agent 拦截 + 确认是否拿对 | 防止拿错件 |

## 第四章 第一层 Agent 详细设计

### 4.1 Agent-1：库存管家（Inventory Steward）

这是整个系统的核心 Agent，是二级库的管控大脑。

#### IDENTITY

```yaml
name: inventory-steward
domain: 备件管理/库存管控
core_responsibility: |
  二级备件库的管控大脑。
  在每一次备件领用中注入判断力：该不该拿、拿什么、拿多少、拿了之后跟踪闭环。
  同时是整个 Agent 网络的数据源头——每次交互都在为未来的智能能力积累数据。

decision_authority: AUTO_WITH_CONFIRM
risk_level: MEDIUM

i_do:
  - 理解自然语言描述的领用需求，提取故障现象、关联设备、所需备件
  - 校验领用合理性：历史消耗对比、频次异常检测、型号匹配校验
  - 推荐最优领用方案：调用替代件能力、引导消化呆滞库存
  - 生成结构化领用指令（通道A）或异步补全出库记录（通道B）
  - 高风险领用拦截（>2000元、安全库存见底、重复领用异常）
  - 领用后闭环跟踪：未消耗提醒、退库引导

i_dont:
  - 不替代扫码的物理确认动作
  - 不直接修改 ERP 数据（通过数据中台接口）
  - 不做超过 3 个月的长期预测（留给 Agent-7）
  - 不做安全库存规则的调整建议（留给 Agent-9a）
```

#### 能力矩阵

| 能力 | 确定性部分（代码） | 不确定性部分（LLM） | Loom 映射 |
|---|---|---|---|
| 需求理解 | 物料编码解析、设备编号匹配 | 从自然语言提取故障现象、推断所需备件 | Step（LLM调用） |
| 合理性校验 | 历史消耗查询、频次统计、阈值比较 | 判断频次异常是否有合理解释 | Step（代码计算） |
| 方案推荐 | 库存查询、价格比较、BOM匹配 | 综合多因素给出最优建议、生成自然语言说明 | Step（LLM+代码混合） |
| 指令生成 | 结构化领用单生成、字段填充 | — | Step（代码） |
| 异常拦截 | 规则匹配（价格>2000、库存见底等） | 生成拦截说明、建议替代方案 | Router（条件判断） |
| 闭环跟踪 | 定时检查领用-消耗匹配、未退库检测 | 生成提醒消息 | Step（定时任务） |

#### 通道 A 交互流程（Agent 驱动）

```
┌─────────────────────────────────────────────────────────────────┐
│ 工程师："3号线电机抖得厉害，我要换轴承"                          │
├─────────────────────────────────────────────────────────────────┤
│ Agent 内部处理：                                                 │
│  1. [LLM] 意图提取：故障维修 / 3号线电机 / 振动异常 / 轴承      │
│  2. [代码] 查询：3号线电机 BOM → 适配轴承型号 6205              │
│  3. [代码] 查库存：6205 库存 2个，安全库存 2个                   │
│  4. [代码] 查历史：本月该设备已领用 6205 × 2 次                  │
│  5. [代码] 查呆滞：6205-2RS 有 1 个，库龄 14 个月（呆滞件）      │
│  6. [代码] 查价格：6205 = 200元，6205-2RS = 180元                │
│  7. [LLM] 综合判断 → 生成建议                                   │
├─────────────────────────────────────────────────────────────────┤
│ Agent 回复：                                                     │
│                                                                  │
│  ⚠️ 注意：3号线电机本月已经第3次更换轴承，频次异常。              │
│  建议先检查电机底座对中和润滑状况，排除系统性问题。               │
│                                                                  │
│  如果确认需要更换：                                               │
│  ▸ 推荐方案：领用 6205-2RS（密封型），单价 180元                  │
│    原因：库里有 1 个已放置 14 个月（呆滞件），密封型更适合         │
│    多粉尘环境，优先消化可降低呆滞金额。                           │
│  ▸ 备选方案：领用 6205（标准型），单价 200元                      │
│    注意：领用后库存降至安全线以下，将自动触发采购预警。            │
│                                                                  │
│  确认领用哪个方案？                                               │
├─────────────────────────────────────────────────────────────────┤
│ 工程师："用推荐的那个"                                           │
├─────────────────────────────────────────────────────────────────┤
│ Agent：                                                          │
│  已生成领用指令：                                                 │
│  · 物料：6205-2RS 密封型轴承 × 1                                 │
│  · 库位：二级库A-3号架                                            │
│  · 领用原因：故障维修 - 3号线电机振动异常                         │
│  · 金额：180元                                                   │
│  请前往取件并扫码确认。                                           │
├─────────────────────────────────────────────────────────────────┤
│ [工程师去货架取件，扫码确认]                                      │
│ [系统校验：扫码物料 = 指令物料 ✓]                                 │
│ [出库记录生成，库存更新，呆滞件标记已消化]                        │
└─────────────────────────────────────────────────────────────────┘
```

在这个交互中，Agent **一次对话完成了 6 件事**：

1. 记录了故障原因（振动异常）——未来 Agent-7 的训练数据
2. 关联了设备（3号线电机）——BOM 关系补全
3. 检测了异常频次——可能发现系统性故障
4. 引导消化了呆滞库存——降低呆滞金额
5. 推荐了更优方案——成本节省 20 元 + 更适合工况
6. 生成了完整的结构化记录——数据质量远高于手填表单

#### 通道 B 交互流程（扫码快速 + 异步补全）

```
┌─────────────────────────────────────────────────────────────────┐
│ [紧急故障] 工程师直接去二级库，扫码取走 6205 轴承 × 1            │
│ [系统记录：谁/什么/几个/什么时间 ✓]                              │
│ [领用原因：空  关联设备：空  故障描述：空]                        │
├─────────────────────────────────────────────────────────────────┤
│ ~15 分钟后，Agent 主动发起异步消息：                              │
│                                                                  │
│  "刚才领了 1 个 6205 轴承，帮你补全一下记录：                     │
│   · 是 3 号线电机的故障换件吗？                                   │
│   · 故障现象是什么？"                                             │
│                                                                  │
│  [是，3号线]  [是，其他设备]  [不是故障，是保养]                   │
├─────────────────────────────────────────────────────────────────┤
│ 工程师点选 "是，3号线" 并补充 "电机抖动"                         │
├─────────────────────────────────────────────────────────────────┤
│ Agent 补全记录：                                                 │
│  · 领用原因：故障维修                                             │
│  · 关联设备：3号线电机                                            │
│  · 故障描述：电机振动异常                                         │
│                                                                  │

└─────────────────────────────────────────────────────────────────┘
```

通道 B 的交互极轻：一次点选 + 几个字的补充。不影响紧急操作，但保证了数据完整性。

#### 进化路径

| 时间 | Agent-1 的能力 | 原因 |
|---|---|---|
| Day-1 ~ 第 1 月 | 库存查询、基础合理性校验（价格拦截、库存见底预警） | 仅依赖 ERP 同步数据和备件主数据 |

| 第 7 ~ 12 月 | 根因分析建议、预测性领用提醒 | 积累了故障描述和消耗趋势数据 |

**核心机制：Agent-1 每天都在变聪明，因为它的每一次交互都是训练数据。**

### 4.2 Agent-2：金额看板员（Financial Tracker）

#### IDENTITY

```yaml
name: financial-tracker
domain: 备件管理/金额分析
core_responsibility: |
  库存金额的全景分析师。
  将一级库（T-1 数据中台同步）和二级库（实时扫码数据）的金额汇聚为
  管理者可直接使用的决策视图。直击"总库存金额降低30%"的考核目标。

decision_authority: ADVISORY
risk_level: LOW
```

#### 输出物

| 输出 | 计算方式 | 数据来源 | 更新频率 |
|---|---|---|---|
| 辅料/备件总数量与金额 | SUM(currentQty × unitPrice) per warehouse | 数据中台 + 扫码 | 日更新 |
| 分部门/工段/工序金额 | 按组织维度聚合 | 同上 | 日更新 |
| 领用金额/数量 Top3 | 按财务月聚合排序 | StockMovement 记录 | 日更新 |
| 当前金额 vs 目标金额 | 当前 vs 800万×70% = 560万 | 快照对比 | 日更新 |
| 环比分析 | 本月 vs 上月 | InventorySnapshot（月度快照） | 月更新 |
| 同比分析 | 本月 vs 去年同期 | InventorySnapshot（月度快照） | 月更新 |

#### LLM 能力：数据翻译

Agent-2 的确定性计算部分很简单（加总、排序、环比公式），LLM 的价值是**把数字翻译成管理语言**：

```
数字：本月辅料工段领用金额 152,000 元，环比上升 18.3%

LLM 翻译：
  "本月辅料工段领用金额 15.2 万元，环比上升 18%。
   主要增量来自变频器更换（3 台，共 8.4 万元），集中在 3 号线。
   如果 3 号线变频器更换频次维持当前水平，预计本季度该项领用
   将超出上季度 40%。建议关注 3 号线变频器故障根因。"
```

#### 与其他 Agent 的协作

- **← Agent-1：** 接收实时出入库数据，特别是通道 A 中已标注原因和设备的高质量数据
- **← Agent-9a：** 接收日频优化建议，在看板上展示"如果执行优化建议，金额可降至 XX 万"
- **→ 工段长周会：** 提供会议需要的金额分析报告

### 4.3 Agent-3：替代件顾问（Substitute Advisor）

#### IDENTITY

```yaml
name: substitute-advisor
domain: 备件管理/备件匹配
core_responsibility: |
  备件规格专家。当备件缺货、呆滞需处置、或有成本优化空间时，
  基于规格参数推荐可替代型号。

decision_authority: ADVISORY
risk_level: MEDIUM
```

#### 两种调用场景

**场景 1：Agent-1 实时调用（嵌入领用流程）**

```
Agent-1 发现：工程师要领 X 型号，库里还有兼容的 Y 型号且更便宜（或 Y 是呆滞件）
  → 调用 Agent-3：确认 X 和 Y 的规格兼容性
  → Agent-3 返回：兼容/部分兼容（附条件说明）/不兼容
  → Agent-1 据此向工程师推荐
```

**场景 2：Agent-4 批量调用（呆滞处置）**

```
Agent-4 识别出 50 个呆滞备件
  → 批量调用 Agent-3：为每个呆滞件查找可替代的在用型号
  → Agent-3 返回：可替代列表 + 兼容度评分
  → Agent-4 生成处置方案："呆滞件 A 可替代在用件 B，建议下次 B 的领用改用 A"
```

#### 匹配逻辑

| 层级 | 匹配方式 | 确定性 |
|---|---|---|
| 精确匹配 | 物料编码直接映射的替代件关系（台账中维护） | 代码 |
| 参数匹配 | 尺寸、额定值、接口类型、工作环境等参数对比 | 代码 |
| 模糊匹配 | 参数不完全一致但在可接受范围内（如额定电流差 10%） | LLM 判断 |
| 文档匹配 | 从技术文档、图纸中 NLP 提取规格信息 | LLM |

### 4.4 Agent-4：呆滞库存侦探（Stale Inventory Detective）

#### IDENTITY

```yaml
name: stale-inventory-detective
domain: 备件管理/呆滞管理
core_responsibility: |
  呆滞库存的发现者和处置策划师。
  按规则标记呆滞备件，分析呆滞原因，生成分级处置方案。

decision_authority: AUTO（标记）+ ADVISORY（处置建议）
risk_level: LOW
```

#### 识别规则

```
呆滞标记规则（当前确认版）：
  IF 库龄 > 12 个月
  THEN 标记为呆滞

后续可扩展：
  IF 库龄 > 12 个月 AND 质保期已过  → 高呆滞（优先处置）
  IF 库龄 > 12 个月 AND 质保期未过  → 低呆滞（可保留观察）
  IF 库龄 > 6 个月 AND 设备已淘汰    → 确定呆滞（必须处置）
```

#### 处置建议生成（LLM 能力）

Agent-4 不只是标记，它分析**为什么呆滞**并推荐处置路径：

| 呆滞原因 | LLM 推断依据 | 推荐处置 |
|---|---|---|
| 设备淘汰 | 该备件适配的设备已不在设备台账中 | 报废 / 退回供应商 |
| 过度采购 | 采购量远超同期消耗量 | 调拨 / 折价处理 |
| 规格变更 | 新采购同类备件为不同型号 | 代用匹配（调用 Agent-3） |
| 偶发故障件 | 故障频率极低但采购周期长、关键性高 | **豁免呆滞**（这类件应备不应清） |

最后一条尤其重要：不是所有呆滞件都该处置。进口 PLC 模块可能两年没动，但一旦需要而没有就是停机——Agent-4 要有判断力识别这种"战略储备"和真正的"浪费"。

#### 主动嵌入领用流程

Agent-4 不是被动等人来查呆滞清单。它通过 Agent-1 嵌入日常领用：

```
Agent-1 收到领用需求：6205 轴承
  → Agent-1 调用 Agent-4：有没有可消化的呆滞件？
  → Agent-4 检查：6205-2RS 库龄 14 个月，规格兼容（已由 Agent-3 确认）
  → Agent-1 向工程师推荐："建议优先使用 6205-2RS，可消化呆滞库存"
```

**每一次领用都是一次消化呆滞的机会。**

### 4.5 Agent-9a：日频库存优化建议（Daily Optimizer）

#### IDENTITY

```yaml
name: daily-optimizer
domain: 备件管理/库存优化
core_responsibility: |
  每日基于当天出库数据和当前库存状态，计算"哪些备件可以降库存、
  降多少、能省多少钱"，直接服务"总库存金额降低30%"的考核目标。

decision_authority: ADVISORY
risk_level: LOW
```

#### 计算逻辑（纯确定性，客户已定义）

```
输入：
  1. 当天领料单数据（数据中台）
  2. 当前库存的备件金额与数量

计算：
  FOR EACH 备件 IN 当前库存：
    当前库存数量
    当前安全库存（默认 = 2）
    近30天日均消耗量
    采购周期（历史平均）
    建议安全库存 = CEIL(日均消耗 × 采购周期 × 1.5)
    可降数量 = MAX(0, 当前库存 - 建议安全库存)
    可降金额 = 可降数量 × 入库标准价

输出：
  1. 备件建议清单（按可降金额降序）
  2. 当前库存可降总数量
  3. 当前安全库存可降数量
  4. 可降低的总金额
  5. 如果全部执行，库存金额将从 X 万降至 Y 万（距30%目标还差 Z 万）
```

#### LLM 增值层

确定性计算输出的是数字表格。LLM 把它变成**管理者能直接决策的建议**：

```
计算结果：可降总金额 48 万元，涉及 23 种备件

LLM 翻译：
  "今日优化建议（截至 3 月 19 日）：

   当前一级库存金额 812 万元，距目标 560 万元还差 252 万元。

   建议优先处理的 3 项：
   1. 6205 轴承：当前库存 30 个，建议降至 12 个，释放 3,600 元
      理由：月均消耗 4 个，采购周期 14 天，12 个可支撑 3 个月。
   2. X 型变频器：当前库存 8 个，建议降至 3 个，释放 25,000 元
      理由：月均消耗 0.5 个，采购周期 30 天，3 个可支撑 6 个月。
   3. ……

   ⚠️ 以下备件建议维持当前库存不降：
   · 进口 PLC 模块（2个）：虽然 12 个月无消耗，但关键性极高、采购周期 90 天，
     属于战略储备。

   如果以上建议全部执行，库存金额将降至 764 万元，
   完成目标的 19%（252 万缺口的 48 万）。"
```

#### 与 What-If 推演的关系

Agent-9a 是**轻量版 What-If**——它回答的是"基于当前数据，什么可以降"。

未来的 Agent-9b（场景推演师）回答的是"如果某个假设条件变化，会怎样"——比如"如果 3 号线换了新电机，轴承消耗率降一半，库存可以降多少"。Agent-9b 需要预测模型支撑，Agent-9a 只需要算术。

## 第五章 第二层 Agent 设计

### 5.1 Agent-5：采购建议师（Procurement Advisor）

#### 解锁条件

| 条件 | 预计时间 | 说明 |
|---|---|---|
| A/B/C 分类规则共创完成 | 第 2-3 月 | 与客户共创备件关键性分级 |
| 二级库有 3 个月出入库数据 | 第 4 月 | Agent-1 积累 |
| 至少一轮采购闭环数据 | 第 4-5 月 | 从请购到到货的完整周期 |
| 通威总部 ERP 采购接口可用 | 需协调 | 跨组织接口，需与@李嘉伟对接 |

#### 核心能力

```
每月 3 号和 18 号（采购节点 5 号/20 号前 2 天）自动生成请购建议：

FOR EACH 库存头寸 WHERE 可用量 < 安全库存 × 1.2：
  计算缺口 = 安全库存 - (当前可用 + 在途量 - 预留量)
  评估紧急度 = 缺口 / 日均消耗 → 几天后真正断货
  生成建议：买什么、买多少、建议何时下单、预计花多少钱

LLM 增值：
  - 采购优先级排序推理（"A 关联的设备本月有大保养，优先采购 A"）
  - 生成给工段长审批的自然语言请购说明
  - 汇总本批次采购总金额，与金额管控目标比对
```

#### 审批流程（Loom 暂停-恢复）

```
Agent-5 生成请购建议清单
  → Loom 暂停，等待工段长审批
  → 工段长在 Weave 界面上 确认 / 修改数量 / 驳回某项
  → Loom 恢复，Agent-5 处理审批结果
  → 确认项 → 生成采购申请 → 推送至 ERP 接口
  → 驳回项 → 记录驳回原因 → 写入 DecisionLog
```

### 5.2 Agent-6：预留协调员（Reservation Coordinator）

#### 解锁条件

| 条件 | 预计时间 | 说明 |
|---|---|---|
| 保养方案模板建立 | 第 3-6 月 | 帮客户定义"某设备某级保养需要哪些备件" |
| Agent-1 积累的保养领用数据 | 第 6 月 | 从保养领用记录反推保养-备件对应关系 |

#### 核心能力

```
保养计划发布
  → Agent-6 查保养方案模板 → 计算所需备件清单
  → 检查库存可用量
  → IF 可用量足够：锁定预留量，可用量扣减
  → IF 可用量不足：通知 Agent-5 生成加急采购建议
  → 保养完成 / 取消 / 超时：自动释放预留量
```

## 第六章 第三层 Agent 设计

### 6.1 Agent-7：需求预测师（Demand Forecaster）

**解锁条件：** 12 个月二级库数据 + 故障-备件关联数据 + 保养-备件关联数据

**核心输出：** 未来 1/3/6 个月各备件需求量预测、建议采购量

**模型来源：** 由 Agent-1 在 12 个月内积累的结构化数据训练。关键数据包括：
- 消耗时序（Agent-1 记录的每次出库）
- 消耗场景标签（故障/保养/技改——Agent-1 在对话中自然采集）
- 设备-备件关联（Agent-1 在领用中关联）
- 季节性信号（12 个月数据足以捕捉年度周期）

### 6.2 Agent-8：设备健康顾问（Equipment Health Advisor）

**解锁条件：** 故障记录电子化 + IoT 数据对接 + 6 个月设备运行数据

**核心输出：** 设备健康度评分、剩余寿命预测（RUL）、提前备件预警

**与备件管理的联动：** 设备健康度下降 → 相关备件的安全水位调高 → 可能触发采购

### 6.3 Agent-9b：场景推演师（Scenario Simulator）

**解锁条件：** Agent-5/7 稳定运行 + 规则引擎参数校准

**核心输出：**
- "如果安全库存从 2 降到 1，缺货风险增加多少？"
- "如果 3 号线故障率翻倍，库存金额会增加多少？"
- "如果开通跨工厂调拨，库存可以降多少？"

**与 Agent-9a 的区别：** 9a 基于事实数据做算术，9b 基于假设条件做模拟。9a 是"今天的优化"，9b 是"策略级决策支持"。

## 第八章 技术架构映射

### 8.1 映射到 Loom 五原语

| Loom 原语 | 在备件 Agent 系统中的体现 |
|---|---|
| **State** | 每个 Agent 的会话状态（领用上下文、查询结果缓存、审批等待状态） |
| **Step** | 每个能力单元（库存查询、合理性校验、LLM 推理、出库单生成）均为一个 Step |
| **Router** | Agent 内的条件分支（通道 A/B 选择、高风险拦截判断、是否调用替代件能力） |
| **Store** | 检查点持久化（长时间等待审批的采购流程、跨日的闭环跟踪） |
| **Graph** | Agent-1 的完整领用处理流程 = 一个 Graph；Agent-5 的采购建议流程 = 一个 Graph |

### 8.2 映射到 Weave 平台

| Weave 组件 | 在备件系统中的体现 |
|---|---|
| **Console** | 工程师的领用对话界面、工段长的审批界面、管理层的看板界面 |
| **RunTimeline** | 采购建议生成过程的可视化（哪些 Agent 参与、每步结果） |
| **SystemMessage** | Agent 异步补全提醒、高风险拦截通知、采购审批通知 |
| **WorkflowSpec** | 每个 Agent 的流程定义（YAML 配置，由 CompileWorkflow 编译为 Graph） |

### 8.3 Agent-Ontology 交互模式

```
Agent 操作 Ontology 的方式：

READ（查询）：
  Agent-1 查库存 → 查询 InventoryPosition.currentQty WHERE warehouse = "二级库A"
  Agent-4 查呆滞 → 查询 InventoryPosition WHERE 库龄 > 12个月
  Agent-3 查替代 → 查询 SparePart.substitutedBy WHERE sparePart = "6205"

WRITE（写回）：
  Agent-1 出库 → 创建 StockMovement 记录 + 更新 InventoryPosition.currentQty
  Agent-4 标记 → 更新 InventoryPosition.isStale = true
  Agent-5 请购 → 创建 PurchaseOrder 对象（状态=待审批）

DERIVE（派生计算）：
  Agent-2 金额 → 触发 InventoryPosition.inventoryValue 重算
  Agent-9a 优化 → 基于 InventoryPosition 集合做聚合计算
```

所有 WRITE 操作都通过 Ontology 层的 **Action** 定义执行，不是 Agent 直接操作数据库。Action 封装了权限校验、写回逻辑和审计日志。

## 第十章 风险与约束

### 10.1 技术风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| LLM 推荐了错误的替代件 | 工程师拿了不兼容的备件，可能损坏设备 | Agent-3 推荐结果必须标注置信度；关键设备的替代件需人工确认；参数匹配有硬约束（电压/电流/尺寸不可超范围） |
| Agent-1 的异常检测误报过多 | 工程师忽略提醒，"狼来了"效应 | 初期只做高确信度异常检测（同一设备同一件3次以上）；逐步调整阈值 |
| 通道 B 异步补全率低 | 工程师不回复补全请求，数据完整率下降 | 追问方式极简化（点选为主，打字为辅）；15 分钟窗口不回复，1 天后再提醒一次；统计完成率，低于 60% 时调整策略 |

### 10.2 业务风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 工程师抵触 Agent 管控 | 绕过系统直接拿备件 | 第一阶段以"建议"而非"审批"模式运行；展示 Agent 带来的实际价值（找到更便宜的替代件、发现频次异常避免停机）；不增加操作步骤——通道 A 替代的是"填表单"不是"多做一件事" |
| 30% 金额降幅目标过激 | 降库存导致缺货停机 | Agent-9a 的优化建议区分"可安全降低"和"建议维持"；关键备件（PLC 模块等）明确标注为"战略储备"不纳入降幅计算 |
| 数据中台接口不稳定 | 一级库数据同步中断 | 本地缓存 T-1 快照；同步失败时在界面明确标注"数据截止至 XX 日"；Agent-2 看板标注数据时效性 |

### 10.3 项目约束

| 约束 | 应对 |
|---|---|
| 一级库数据 T-1（昨日） | Agent 明确标注数据时效；二级库数据为实时 |
| 采购接口在通威总部 | Agent-5 第一阶段先生成请购清单导出，人工在 ERP 操作；接口打通后自动化 |
| 呆滞定义未内部对齐 | Agent-4 第一阶段用最宽松定义（>12月），规则参数可调 |
| 二级库扫码方案尚未实施 | Agent 系统设计不依赖扫码——通道 A 本身就能完成数据录入 |

│   │   ├── [代码] 查历史：同设备同备件最近领用记录
│   │   ├── [代码] 频次检查：本月领用次数 vs 历史月均
│   │   ├── [代码] BOM 校验：备件是否适配该设备
│   │   └── IF 异常 → [LLM] 生成异常提醒 + 建议
│   │
│   ├── Step 3: 方案推荐

│   │   ├── IF 单价 > 2000 → 强制确认 + 提示退库报废流程
│   │   ├── IF 领后库存 < 安全库存 → 警告 + 自动触发采购预警

│   │
