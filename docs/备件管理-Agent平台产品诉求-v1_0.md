# 备件管理 Agent 系统对底层平台的产品诉求

**inocube 智能平台 · Agent 平台能力需求**

v1.0 · 2026.03

---

> *本文档不讨论技术实现。只回答一个问题：*
> *要让备件管理的 9 个 Agent 真正跑起来，底层平台必须具备哪些产品能力？*
> *每条诉求都附带业务场景说明——为什么需要这个能力、没有会怎样。*

---

## 一、Agent 生命周期管理

### 诉求 1.1：Agent 的定义与注册

平台需要能定义一个 Agent 的完整身份信息：它叫什么、负责什么、能做什么、不能做什么、谁可以跟它交互。支持以配置文件（而非硬编码）的方式定义和修改。

**业务场景：** 备件系统有 9 个 Agent，各自职责不同。库存管家能操作出入库，金额看板只能读取数据不能修改。平台需要在定义层面就区分清楚每个 Agent 的能力边界，而不是靠代码里的 if-else 控制。

### 诉求 1.2：Agent 的版本管理

Agent 的定义（职责、能力、规则）会随业务演进而变化。平台需要支持 Agent 定义的版本化——能回滚到上一个版本，能对比两个版本的差异。

**业务场景：** Agent-4（呆滞侦探）的呆滞规则从"库龄>12个月"调整为"库龄>12个月且质保过期"。调整后发现标记数量骤降，需要回滚到上一版规则重新评估。

### 诉求 1.3：Agent 的灰度上线

新 Agent 或 Agent 的新版本能先对部分用户/部分库房生效，验证无问题后再全量推开。

**业务场景：** Agent-1（库存管家）要上线"替代件推荐"新能力。先在一个二级库试运行两周，观察推荐准确率和工程师接受度，再推广到全部三个二级库。

## 二、对话与交互

### 诉求 2.1：多轮对话能力

Agent 必须能进行多轮对话，记住上下文。不是每次都从头开始。

**业务场景：** 工程师说"3号线电机抖动，要换轴承"，Agent 问"是6205还是6206？"，工程师说"6205"。这三轮对话是一个连续的领用意图，Agent 必须记住前面的上下文（3号线、电机、抖动）才能正确生成领用指令。

### 诉求 2.2：主动发起对话

Agent 不能只被动等用户说话。它需要能主动向用户推送消息。

**业务场景：** 通道B——工程师紧急扫码出库后，Agent 需要在15分钟内**主动发起**一条消息："刚才领了1个6205轴承，是3号线电机的故障换件吗？"如果平台不支持 Agent 主动发消息，通道B的异步补全就无法实现。

### 诉求 2.3：快捷回复（结构化选项）

对话中 Agent 提出的问题，需要能以**可点击的选项**形式呈现，而不是只能打字回复。

**业务场景：** Agent 异步补全时问"这次领用是什么原因？"，给出 [故障维修] [保养更换] [技改] [项目实施] 四个按钮。工程师一点就完成，不需要打字。这直接影响通道B的补全完成率——需要打字的补全率可能30%，点选的补全率可能80%。

### 诉求 2.4：多 Agent 之间的会话可见性

当 Agent-1 调用 Agent-3 的能力时，调用过程和结果对用户应该是自然衔接的，而不是弹出一个新的对话窗口。

**业务场景：** 工程师跟 Agent-1 说要领轴承，Agent-1 发现有呆滞兼容件，调用 Agent-3 确认规格兼容性。整个过程在用户看来是一次连续的对话，不是"请切换到替代件顾问"。

## 三、流程编排

### 诉求 3.1：流程的条件分支

Agent 的处理流程中需要支持条件判断——走不同的路径。

**业务场景：** Agent-1 收到领用请求后，需要判断：备件单价是否>2000？库存领用后是否低于安全线？本月该设备是否已领用同一备件3次以上？不同的判断结果走不同的处理流程（正常出库 / 高风险确认 / 异常提醒）。

### 诉求 3.2：Agent 之间的调用

一个 Agent 能在自己的流程中调用另一个 Agent 的能力，拿到结果后继续自己的流程。

**业务场景：** Agent-1 在推荐领用方案时，需要调用 Agent-3 查替代件、调用 Agent-4 查呆滞件。这两个调用应该是 Agent-1 流程的一个步骤，不需要人工介入切换。

### 诉求 3.3：暂停与恢复

Agent 流程执行到某一步时，能暂停等待外部事件（人工审批、系统回调），事件到达后从暂停点继续执行，不丢失任何上下文。

**业务场景：** Agent-5 生成了一批采购建议，需要等工段长审批。工段长可能当天下班前才看，也可能第二天才处理。Agent-5 必须能暂停在"等待审批"状态，审批完成后从断点继续——把确认的项推送到ERP采购接口，把驳回的项记录原因。不能因为等了一晚上就丢失了整个建议清单的上下文。

### 诉求 3.4：定时触发

Agent 流程能被定时任务触发，而不只是被用户对话触发。

**业务场景：**
- Agent-9a（日频优化）需要每天下午 5 点自动运行一次，基于当天的出库数据生成优化建议。
- Agent-5（采购建议）需要每月 3 号和 18 号自动生成请购清单。
- Agent-4（呆滞侦探）需要每月 1 号扫描一次全部库存，更新呆滞标记。

这些都不是用户主动发起的，是系统自动触发的。

### 诉求 3.5：异步消息与事件驱动

Agent 能监听外部事件并做出响应——不是轮询，是事件驱动。

**业务场景：**
- 扫码系统产生了一条出库记录 → Agent-1 收到事件 → 判断是否需要异步补全。
- 数据中台完成了每晚的数据同步 → Agent-2 收到事件 → 更新金额看板数据。
- ERP 反馈采购单已到货 → Agent-5 收到事件 → 闭环采购流程。

## 四、工具与外部系统集成

### 诉求 4.1：工具注册与调用

Agent 需要能调用外部工具（查询数据库、调用API、执行计算），平台需要提供统一的工具注册和调用机制。

**业务场景：** Agent-1 在处理领用请求时需要调用多个工具：查库存数量（数据库查询）、查历史领用记录（数据库查询）、校验BOM匹配（规则引擎）、计算金额（计算函数）。这些工具需要预先注册，Agent 按需调用。

### 诉求 4.2：工具的权限控制

不同 Agent 能调用的工具应该有权限区分。

**业务场景：** Agent-1 能调用"创建出库记录"工具，Agent-2 不能——它只有读取权限。Agent-5 能调用"提交采购申请"工具，其他 Agent 不能。平台需要在工具层面做权限隔离，防止 Agent 越权操作。

### 诉求 4.3：工具调用的审计日志

每一次 Agent 调用工具的行为都要被记录：谁调用了什么、什么时间、传了什么参数、返回了什么结果。

**业务场景：** 如果 Agent-1 生成了一条出库记录后工程师说"我没领这个"，需要回溯完整链路：Agent 是基于什么对话上下文做的判断、调用了哪些工具、传了什么参数。这是审计和纠错的基础。

---

## 五、状态与记忆

### 诉求 5.1：会话级状态

单次对话过程中，Agent 能记住之前说了什么、做了什么。

**业务场景：** 工程师跟 Agent-1 说了3轮话，Agent 生成了领用指令，工程师去扫码——扫码失败（物料不匹配），回来跟 Agent 说"扫码报错了"。Agent 必须记住这是同一次领用会话，能直接理解"报错"指的是刚才的领用指令。

### 诉求 5.2：跨会话记忆

Agent 能记住与某个用户的历史交互——至少是近期的关键信息。

**业务场景：** 工程师上周跟 Agent-1 说过"3号线电机最近不太稳"。这周又来领轴承，Agent 能关联上周的上下文，更准确地判断领用合理性。没有跨会话记忆，Agent 每次都是"新认识你"，无法做历史关联分析。

### 诉求 5.3：Agent 间的共享状态

不同 Agent 能读取彼此产生的关键状态信息。

**业务场景：** Agent-4 标记了一批呆滞备件清单。Agent-1 在处理领用请求时需要读到这个清单，才能在领用环节推荐消化呆滞件。如果 Agent 之间的状态完全隔离，Agent-1 不知道哪些件是呆滞的。

## 六、人机协作

### 诉求 6.1：人工确认环节

Agent 流程中能插入"等待人工确认"的节点。确认、修改、驳回三种操作都要支持。

**业务场景：** Agent-5 生成的采购建议包含 15 条请购项。工段长需要能：确认其中 12 条、把第 13 条的数量从 5 改成 3、驳回第 14 和 15 条并填写理由。平台必须支持这种"逐项审批+修改"的能力，不能只有"全部通过/全部驳回"。

### 诉求 6.2：人工接管

当 Agent 处理不了或判断不确定时，能转给人工处理——人工处理完后，流程可以继续。

**业务场景：** Agent-1 遇到一个从未见过的备件型号，无法判断跟哪台设备适配。它应该能标记"我不确定"，把问题转给设备工程师确认。工程师确认后，Agent 拿到结果继续后续流程。

### 诉求 6.3：决策可解释性

Agent 的每一个建议和决策，用户要能看到**为什么**——推理过程、使用了什么数据、依据了什么规则。

**业务场景：** Agent-9a 建议"6205轴承库存从30降到12"。工段长需要看到：月均消耗4个、采购周期14天、1.5倍安全系数、所以安全库存=4×14/30×1.5≈3，加上一个采购周期的覆盖=3+4×14/30≈5，考虑波动取12。如果工段长看不到这个推理链，他不会信任这个建议。

---

## 七、安全与权限

### 诉求 7.1：操作级权限

不同 Agent 能执行的操作类型要有权限控制：只读、建议、需确认后执行、自动执行。

**业务场景：**
- Agent-2（金额看板）：只读，不能修改任何数据。
- Agent-4（呆滞侦探）：标记呆滞 = 自动执行；处置建议 = 仅建议。
- Agent-1（库存管家）：生成领用指令 = 需人工确认后执行。
- Agent-5（采购建议）：生成请购 = 需审批后执行。

### 诉求 7.2：用户级权限

不同角色的用户能跟不同的 Agent 交互、看到不同的信息。

**业务场景：**
- 工程师：能跟 Agent-1 对话（领用）、看到 Agent-2 的工段级看板。
- 工段长：能审批 Agent-5 的采购建议、看到 Agent-2 的全局看板、操作 Agent-9a 的优化建议。
- 管理层：只看 Agent-2 看板和 Agent-9a 报告，不参与日常操作。

### 诉求 7.3：敏感操作拦截

特定条件下 Agent 的操作必须被强制拦截，无论用户怎么说。

**业务场景：** 备件单价>2000元的出库操作，即使工程师确认了，也要走退库报废流程。Agent 不能因为"用户说没问题"就跳过这个管控。平台需要支持不可绕过的硬规则。

---

## 八、可观测性

### 诉求 8.1：Agent 运行状态监控

能看到每个 Agent 当前是否正常运行、有没有卡在某个环节、有多少任务在排队。

**业务场景：** Agent-5 每月 3 号自动生成采购建议。如果 3 号早上跑了但因为数据中台同步异常卡住了，管理员需要能及时发现而不是到 5 号采购截止日才发现建议没生成。

### 诉求 8.2：完整的执行链路追溯

每一次 Agent 的完整执行过程——从用户输入到最终输出，中间经过了哪些步骤、调用了哪些工具、调用了哪些其他 Agent、每一步的输入输出——都要能事后查看。

**业务场景：** 工段长质疑 Agent-9a 的优化建议数字不对。运维人员需要能查到：Agent-9a 这次运行时读取了哪些库存数据、用了什么计算公式、中间结果是什么、最终建议怎么得出的。不能是一个黑盒。

### 诉求 8.3：业务指标统计

平台能统计 Agent 的业务效果数据：处理了多少请求、建议被接受了多少、补全完成率多少。

**业务场景：**
- Agent-1 通道B的异步补全完成率是多少？（低于60%需要调整策略）
- Agent-3 的替代件推荐被工程师接受了多少次？（衡量推荐质量）
- Agent-9a 的优化建议实际执行了多少？（衡量建议可行性）

这些指标是持续优化 Agent 的依据。

## 九、成本控制

### 诉求 9.1：大模型调用的用量可见

每个 Agent 消耗了多少大模型调用量（token / 次数 / 金额），需要能看到和追踪。

**业务场景：** Agent-1 每天处理上百次领用对话，是大模型消耗大户。管理员需要知道每月的 AI 成本是多少，Agent-1 占了多少比例，以此评估投入产出比。

### 诉求 9.2：成本预算上限

能为单个 Agent 或整个系统设置大模型调用的预算上限——超了自动降级或暂停，而不是无限花钱。

**业务场景：** 如果 Agent-1 因为某个 bug 陷入循环对话，不断调用大模型，一天就可能烧掉一个月的预算。平台需要有保护机制。

### 诉求 9.3：按需加载，降低每次调用成本

Agent 有很多能力，但大多数对话只用到其中 1-2 项。平台应该支持只加载当前需要的能力描述给大模型，而不是每次都把所有能力全塞进去。

**业务场景：** Agent-1 有 6 项能力（需求理解、合理性校验、方案推荐、异常拦截、指令生成、闭环跟踪）。工程师问"3号架还有几个6205？"只需要"库存查询"能力，没必要把"异常拦截"和"闭环跟踪"的完整描述也发给大模型。这能显著降低每次调用的 token 消耗。

## 十、部署与扩展

### 诉求 10.1：新 Agent 上线不影响已有 Agent

第二层 Agent（Agent-5/6）上线时，不需要停机或重启第一层 Agent（Agent-1/2/3/4/9a）。

**业务场景：** Agent-1 每天服务几十个工程师的领用请求。3个月后 Agent-5 准备上线，上线过程中 Agent-1 必须正常运行，不能中断日常业务。

### 诉求 10.2：Agent 能力的热更新

Agent 的规则和参数能在不重启的情况下更新生效。

**业务场景：** 呆滞定义从">12个月"改为">12个月且质保过期"。这个规则变更应该能立即生效，不需要停掉 Agent-4 再重启。

### 诉求 10.3：支持逐步解锁 Agent 能力

单个 Agent 的能力可以分阶段开启——Day-1 开启基础能力，3个月后开启进阶能力，不需要换一个 Agent。

**业务场景：** Agent-1 的进化路径：
- Day-1：库存查询 + 基础合理性校验
- 第 3 月：重复领用检测 + 频次异常
- 第 6 月：设备-备件关联推荐
- 第 12 月：根因分析建议

这是同一个 Agent-1 的能力逐步解锁，不是 4 个不同的 Agent 替换上线。

---

## 十一、崩溃恢复

### 诉求 11.1：流程中断后可恢复

如果平台发生故障重启，正在执行中的 Agent 流程能从中断点恢复，而不是从头来过。

**业务场景：** Agent-5 正在处理一批采购建议，已经完成了 10 条中的 7 条的审批确认。此时系统崩溃重启。恢复后应该从第 8 条继续，而不是重新生成整批建议让工段长重新审一遍。

### 诉求 11.2：等待中的流程不丢失

Agent 暂停等待人工操作期间（可能等几小时甚至几天），如果系统重启，等待状态和全部上下文不能丢失。

**业务场景：** Agent-5 生成了采购建议并暂停等工段长审批。工段长出差了，3天后才回来审批。这3天里系统可能经历过维护重启。重启后 Agent-5 必须还在等待状态，审批界面上的建议清单必须完好。

---

## 十二、诉求优先级总结

按实现的紧迫度排序——没有这个能力，对应的业务场景就跑不起来。

### P0：Day-1 上线必须具备

| # | 诉求 | 不具备的后果 |
|---|---|---|
| 2.1 | 多轮对话 | Agent-1 无法理解领用意图 |
| 2.2 | 主动发起对话 | 通道B 异步补全无法实现 |
| 2.3 | 快捷回复选项 | 补全完成率过低 |
| 3.1 | 条件分支 | 高风险拦截无法实现 |
| 3.2 | Agent间调用 | Agent-1 无法调用替代件和呆滞查询 |
| 4.1 | 工具注册与调用 | Agent 无法查询数据库和执行操作 |
| 4.2 | 工具权限控制 | Agent 可能越权操作 |
| 5.1 | 会话级状态 | 多轮对话上下文丢失 |
| 7.1 | 操作级权限 | 无法区分只读/建议/执行 |
| 7.3 | 敏感操作拦截 | 高价值备件管控失效 |
