## 第一章 总述

### 1.1 本体范围

本本体覆盖备件管理的完整生命周期：从备件入库、库存持有、领用消耗、安全预警、采购补货到呆滞处置。不覆盖设备运维（故障诊断、IoT监控）和生产排程。

### 1.2 第一公民

**库存头寸（InventoryPosition）** 是本本体的第一公民。

定义：一种备件在一个具体库房的持有状态。

判断依据：管理者每天打开系统盯着的不是"某种备件总体怎么样"，而是"这个库房里这种备件够不够、值多少钱、放了多久"。所有分析、决策、行动的原子单位都是头寸。

### 1.3 设计分层

| 层 | 内容 | 稳定性 |
|---|---|---|
| **语义层** | 类（对象类型）、属性、关系（链接类型） | 高——跨项目复用 |
| **动能层** | 规则、动作、决策日志 | 中——规则参数可配置 |
| **信号层** | 信号拓扑（什么变化驱动什么决策）、因果传导链 | 中——路径固定，强度可变 |
| **演化层** | MVO-α（最小可用本体·数据底座）→ MVO-β（规则引擎）→ 完整本体 | 变——按数据积累逐步扩展 |

### 1.4 命名规范

| 元素 | 命名 | 示例 |
|---|---|---|
| 类名 | 英文 PascalCase + 中文标注 | InventoryPosition（库存头寸） |
| 属性名 | 英文 camelCase + 中文列标注 | currentQty（当前数量）、unitPrice（入库标准价） |
| 关系名 | 英文动词短语 + 中文标注 | tracks（跟踪）、locatedIn（位于）、affects（影响） |
| 规则名 | R + 两位编号 + 中文名 | R01 安全库存预警 |
| 动作名 | A + 两位编号 + 中文名 | A01 生成请购建议 |

### 2.2 InventoryPosition（库存头寸）—— 第一公民

一种备件在一个具体库房的持有状态。所有分析和决策的原子单位。

#### 属性表

| 属性名 | 中文 | 类型 | 必填 | 来源 | 阶段 | 说明 |
|---|---|---|---|---|---|---|
| **静态身份** | | | | | | |
| id | 头寸ID | string | ✅ | 系统生成 | α | 主键，格式：{warehouseId}_{sparePartId} |
| sparePartRef | 备件引用 | → SparePart | ✅ | 台账 | α | 该头寸跟踪哪种备件 |
| warehouseRef | 库房引用 | → Warehouse | ✅ | 台账 | α | 该头寸在哪个库房 |
| storageLocation | 仓位位置 | string | — | 台账 | α | 货架位置（如"A-3号架"） |
| **动态状态** | | | | | | |
| currentQty | 当前数量 | int | ✅ | 数据中台同步/扫码 | α | 一级库T-1同步，二级库实时 |
| reservedQty | 预留数量 | int | — | 系统计算 | β | 被保养计划锁定的数量 |
| availableQty | 可用数量 | int(派生) | — | 派生 | α | = currentQty - reservedQty |
| unitPrice | 入库标准价 | decimal | ✅ | 数据中台 | α | 标准价格（入库价格），不变 |
| inventoryValue | 库存金额 | decimal(派生) | — | 派生 | α | = currentQty × unitPrice |
| safetyStock | 安全库存 | int | ✅ | 规则/人工 | α | 初始值=2（全局默认），后续可差异化 |
| safetyGap | 安全缺口 | int(派生) | — | 派生 | α | = safetyStock - availableQty，正值=缺货风险 |
| stockAge | 库龄 | int(派生) | — | 派生 | α | 最早批次的入库天数 |
| isStale | 是否呆滞 | boolean(派生) | — | 规则R03 | α | 库龄>365天 |
| lastMovementDate | 末次变动日期 | date | — | 出入库记录 | α | 最近一次出库或入库的日期 |
| dataSource | 数据来源 | enum | ✅ | 系统 | α | ERP_SYNC（数据中台同步）/ SCAN_REALTIME（扫码实时） |
| dataAsOf | 数据截止时间 | datetime | ✅ | 系统 | α | 标注数据时效性 |
| **行为画像**（需数据积累） | | | | | | |
| avgMonthlyConsumption | 月均消耗 | float(派生) | — | 12个月出库聚合 | β | 过去12个月月均出库量 |
| consumptionVolatility | 消耗波动系数 | float(派生) | — | 12个月出库聚合 | β | 标准差/均值 |
| turnoverRate | 周转率 | float(派生) | — | 派生 | β | 年消耗量/平均库存 |
| daysOfSupply | 可支撑天数 | float(派生) | — | 派生 | β | currentQty / 日均消耗 |
| lastConsumedDate | 末次消耗日期 | date | — | 出库记录 | α | 最近一次出库（不含退库）的日期 |

#### 派生属性计算公式

```
可用数量 availableQty       = 当前数量 currentQty - 预留数量 reservedQty
库存金额 inventoryValue     = 当前数量 currentQty × 入库标准价 unitPrice
安全缺口 safetyGap          = 安全库存 safetyStock - 可用数量 availableQty
                               （正值=缺货风险，负值=充裕）
是否呆滞 isStale            = (今天 - 末次消耗日期) > 365天
                               OR (库龄 > 365天 AND 末次消耗日期为空)
月均消耗 avgMonthlyConsumption = SUM(出库数量, 过去12个月) / 12
可支撑天数 daysOfSupply     = 当前数量 / (月均消耗 / 30)
周转率 turnoverRate         = (月均消耗 × 12) / ((期初库存 + 期末库存) / 2)
```

每一次库存变动的记录。是头寸动态状态变化的事实来源。

| 属性名 | 中文 | 类型 | 必填 | 说明 |
|---|---|---|---|---|

| positionRef | 头寸引用 | → InventoryPosition | ✅ | 影响哪个头寸 |
| movementType | 变动类型 | enum | ✅ | IN(入库) / OUT(出库) / RETURN(退库) / SCRAP(报废) / TRANSFER(调拨) |
| movementReason | 变动原因 | enum | ✅ | FAULT_REPAIR(故障维修) / MAINTENANCE(保养) / TECH_UPGRADE(技改) / PROJECT(项目实施) / SCRAP_RETURN(报废退库) / OTHER |

| movementDate | 变动日期 | datetime | ✅ | |
| operatorId | 操作人 | string | ✅ | 谁执行的 |
| warehouseLevel | 库房层级 | enum | ✅ | PRIMARY（一级库）/ SECONDARY（二级库） |
| source | 数据来源 | enum | ✅ | ERP_SYNC(数据中台同步) / SCAN(扫码录入) / AGENT(Agent生成) |
| relatedEquipmentId | 关联设备 | string | — | 领用时关联的设备编号（Agent-1补全） |

- relatedEquipmentId 和 faultDescription 由 Agent-1 在交互中采集或异步补全。这两个字段是 Agent-7（需求预测）未来的训练数据来源。

## 第三章 语义层——关系定义

### 3.1 关系全景

```
Equipment ──uses(使用)──→ SparePart ←──tracks(跟踪)── InventoryPosition ──locatedIn(位于)──→ Warehouse
（设备）                   （备件）                      （库存头寸）                          （库房）
                              ↑                              ↑
                              │                              │
            PurchaseOrder ──procures(采购)──→      StockMovement ──affects(影响)──→
             （采购单）                              （出入库记录）
                              │
              SparePart ──substitutedBy(可替代)──→ SparePart（自关联）
```

### 3.2 关系详细定义

| 关系名 | 中文 | 主体 | 客体 | 多重性 | 阶段 | 业务含义 |
|---|---|---|---|---|---|---|
| **tracks** | 跟踪 | 库存头寸 | 备件 | 1 | α | 头寸跟踪哪种备件 |
| **locatedIn** | 位于 | 库存头寸 | 库房 | 1 | α | 头寸在哪个库房 |
| **affects** | 影响 | 出入库记录 | 库存头寸 | 1 | α | 出入库记录影响哪个头寸 |
| **procures** | 采购 | 采购单 | 备件 | 1 | α | 采购单采购哪种备件 |
| **uses** | 使用 | 设备 | 备件 | 1..* | α | 设备使用哪些备件（BOM关系） |
| **substitutedBy** | 可替代 | 备件 | 备件 | 0..* | β | 备件可被哪些型号替代 |
| **snapshotOf** | 快照于 | 库存快照 | 库存头寸 | 1 | α | 快照记录哪个头寸 |
| **decidesOn** | 决策于 | 决策日志 | 库存头寸 | 0..1 | β | 决策针对哪个头寸 |
| **consumedBy** | 消耗于 | 出入库记录 | 设备 | 0..1 | α | 领用给了哪台设备（Agent-1补全） |

### 3.3 关系约束

| 约束 | 类型 | 说明 |
|---|---|---|
| 一个头寸唯一确定一个备件+库房组合 | 唯一性 | (sparePartRef, warehouseRef) 联合唯一 |
| substitutedBy（可替代）不能自引用 | 依赖 | A.substitutedBy 不能包含 A 自身 |
| PurchaseOrder.status（采购单状态）状态机 | 值域 | 草稿→已提交→已审批→在途→已到货；任意状态→已取消 |
| StockMovement（出入库记录）影响的头寸必须在同一库房 | 一致性 | movement.warehouseLevel（库房层级）必须与 position.warehouse.level 一致 |

| 信号 | 来源 | 可预测性 | 对头寸影响 | 数据状态 |
|---|---|---|---|---|
| **故障维修领用** | StockMovement(reason=FAULT_REPAIR) | 低——只能估概率 | 突然减少 | ✅ 一级库有单据；二级库由Agent-1采集 |
| **保养领用** | StockMovement(reason=MAINTENANCE) | 中——有计划但不含备件清单 | 可预见减少 | ⚠️ 保养计划不含备件清单，退化为不确定性信号 |
| **技改领用** | StockMovement(reason=TECH_UPGRADE) | 中——有项目计划 | 批量减少 | ✅ 有流程 |
| **项目领用** | StockMovement(reason=PROJECT) | 高——有项目计划 | 批量减少 | ✅ 较少 |

### 4.2 补给侧信号

| 信号 | 来源 | 关键参数 | 对头寸影响 | 数据状态 |
|---|---|---|---|---|
| **采购入库** | PurchaseOrder→StockMovement(IN) | 采购周期（2天~180天） | 增加（有时间差） | ✅ 数据中台可获取 |
| **退库** | StockMovement(RETURN) | 退库比例 | 小量增加 | ⚠️ 一级库有记录；二级库不确定 |
| **调拨入库** | StockMovement(TRANSFER) | — | — | ❌ 机制不存在，推迟 |

### 4.3 因果传导链（完整本体阶段）

```
设备健康度下降（Agent-8 设备健康顾问 检测）
  → 故障概率上升
  → 关联备件的预期消耗压力增大
    （通过图谱遍历：设备 ─使用→ 备件 ─跟踪← 库存头寸 ─位于→ 库房）
  → 多个库存头寸的安全水位需调高
  → 库房总金额可能超标
  → 需从过剩头寸释放资金（Agent-9a 日频优化 计算）
```

MVO-α/β 阶段此链条不可用（故障数据不存在）。退化为基于消耗统计的独立头寸分析。

## 第五章 动能层——规则定义

### 5.1 规则总览

| 规则ID | 名称 | 阶段 | 触发方式 | 权限 |
|---|---|---|---|---|
| R01 | 安全库存预警 | α | 自动（库存变动后） | 自动执行 |
| R02 | 低库存采购触发 | α | 自动（R01触发后） | 自动生成+人工确认 |
| R03 | 呆滞标记 | α | 自动（每月1号扫描） | 自动执行 |
| R04 | 高价值领用拦截 | α | 自动（出库时） | 需人工确认 |
| R05 | 频次异常检测 | α | 自动（出库时） | 仅提醒 |
| R06 | 差异化安全库存 | β | 人工配置+系统计算 | 人工发起 |
| R07 | 日频优化计算 | α | 定时（每日17:00） | 仅建议 |
| R08 | 月度快照生成 | α | 定时（每月末日） | 自动执行 |

### 5.2 规则详细定义

#### R01 安全库存预警

```
触发条件：库存头寸的 当前数量（currentQty）发生变化后
计算：  安全缺口 = 安全库存 - 可用数量
判断：  如果 安全缺口 > 0 则触发预警
输出：  预警事件（头寸ID、缺口数量、缺口金额）
写回：  无（纯通知）
联动：  触发 R02（低库存采购触发）
参数：  安全库存（可配置，初始=2）
```

#### R02 低库存采购触发

```
触发条件：R01（安全库存预警）产生预警后
计算：  采购建议量 = 安全缺口 + (月均消耗 × 典型采购周期天数 / 30)
        如果 月均消耗 不可用（数据积累不足），回退公式 = 安全缺口 × 2
判断：  如果已有该备件的在途采购单，则扣减在途量后重算
输出：  采购建议项（备件、数量、建议下单时间、预估金额）
写回：  创建采购单（状态=草稿）
权限：  草稿状态自动生成，提交需工段长确认
```

#### R03 呆滞标记

```
触发条件：每月1号全量扫描所有库存头寸
计算：  逐个检查每个库存头寸：
          如果 末次消耗日期为空 且 库龄 > 365天 → 标记呆滞
          如果 末次消耗日期不为空 且 (今天 - 末次消耗日期) > 365天 → 标记呆滞
写回：  更新库存头寸的 是否呆滞（isStale）字段
参数：  呆滞阈值天数（可配置，初始=365天）
注意：  呆滞定义客户内部尚未完全对齐（纯库龄 vs 库龄+质保期），做成可配置参数
```

#### R04 高价值领用拦截

```
触发条件：创建出库记录（变动类型=出库）时
计算：  关联备件单价 = 头寸.入库标准价
判断：  如果 单价 > 2000元 则拦截
输出：  拦截通知（提示走退库报废流程）
写回：  出入库记录标记为"待确认"（PENDING_CONFIRM）
权限：  需人工确认后才能完成出库
参数：  拦截阈值（可配置，初始=2000元）
```

#### R05 频次异常检测

```
触发条件：创建出库记录（变动类型=出库）时
计算：  统计近30天内 同一设备 + 同一备件 的出库次数
判断：  如果 次数 >= 3 则标记异常
输出：  异常提醒（可能存在系统性故障，建议排查根因）
写回：  无（纯通知，由Agent-1 库存管家呈现给用户）
参数：  异常阈值（可配置，初始=3次/30天）
```

#### R06 差异化安全库存（MVO-β）

```
触发条件：备件的 关键性等级（criticality）被设置后
计算：  关键性=A（关键）→ 安全库存 = MAX(3, 向上取整(月均消耗 × 典型采购周期天数 / 30 × 2.0))
        关键性=B（重要）→ 安全库存 = MAX(2, 向上取整(月均消耗 × 典型采购周期天数 / 30 × 1.5))
        关键性=C（一般）→ 安全库存 = MAX(1, 向上取整(月均消耗 × 典型采购周期天数 / 30 × 1.0))
写回：  更新库存头寸的 安全库存（safetyStock）
前置：  需要 月均消耗 和 典型采购周期 数据可用（3个月以上积累）
```

#### R07 日频优化计算

```
触发条件：每日 17:00 定时触发
输入：  当天所有出入库记录 + 所有库存头寸当前状态
计算：  逐个检查每个库存头寸：
          建议安全库存 = 向上取整(日均消耗 × 典型采购周期天数 × 1.5)
          可降数量 = MAX(0, 当前数量 - 建议安全库存)
          可降金额 = 可降数量 × 入库标准价
输出：  优化建议清单（按可降金额降序）、总可降数量、总可降金额、距目标差距
写回：  无（纯建议，由 Agent-9a 日频优化 呈现）
```

#### R08 月度快照生成

```
触发条件：每月最后一天 23:59
计算：  逐个检查每个库存头寸：
          创建库存快照，记录当前数量和金额
          计算本月净消耗 = 本月出库总量 - 本月退库总量
写回：  创建库存快照（InventorySnapshot）记录
```

### 6.1 动作总览

| 动作ID | 名称 | 阶段 | 触发 | 权限 | 写回目标 |
|---|---|---|---|---|---|
| A01 | 记录出库 | α | Agent-1/扫码 | 自动+确认 | StockMovement + InventoryPosition |
| A02 | 记录入库 | α | 数据中台同步/扫码 | 自动 | StockMovement + InventoryPosition |

### 6.2 动作详细定义

#### A01 记录出库

```yaml
名称: 记录出库
触发: Agent-1（库存管家）生成领用指令（通道A）/ 扫码出库（通道B）
参数:
  - positionId（头寸ID）: 必填
  - quantity（数量）: 必填
  - movementReason（领用原因）: 必填——通道A由Agent对话获取，通道B由Agent异步补全
  - operatorId（操作人）: 必填
  - relatedEquipmentId（关联设备）: 可选——Agent-1在对话中采集
  - faultDescription（故障描述）: 可选——Agent-1在对话中采集
前置校验:
  - 数量 > 0
  - 头寸.当前数量 >= 领用数量（防止出库后为负）
  - 如果 备件单价 > 2000元 则触发 R04（高价值领用拦截）
写回:
  - 创建出入库记录（StockMovement）
  - 库存头寸.当前数量 -= 领用数量
  - 库存头寸.末次变动日期 = 当前时间
  - 触发 R01（安全库存预警）重算安全缺口
  - 触发 R05（频次异常检测）
风险等级: 中
```

#### A04 生成请购建议

```yaml
名称: 生成请购建议
触发: R02（低库存采购触发）/ Agent-5（采购建议师）定时生成（每月3/18号）
参数:
  - positionId（头寸ID）: 可批量
  - suggestedQty（建议采购量）: 系统计算
  - urgency（紧急度）: 常规(NORMAL) / 紧急(URGENT)
  - reasoning（建议理由）: Agent-5生成的自然语言说明
前置校验:
  - 不存在同一备件的"在途"状态采购单（防重复采购）
写回:
  - 创建采购单（状态=草稿，来源=Agent建议）
  - 流程暂停，等待人工审批
审批后:
  - 确认 → 状态变为"已提交" → 推送至通威总部ERP接口
  - 修改 → 更新数量 → 记录决策日志（类型=修改）→ 推送
  - 驳回 → 状态变为"已取消" → 记录决策日志（类型=驳回）
风险等级: 高（涉及资金）
```

## 第九章 存储策略

### 9.1 混合存储

| 数据类型 | 存储选择 | 原因 |
|---|---|---|
| 类定义（TBox，术语层——描述"有哪些概念和规则"） | Neo4j / PostgreSQL JSON | 类和关系的结构定义 |
| 实例数据（ABox，事实层——描述"具体有哪些东西"）| PostgreSQL | 头寸、备件、采购单等结构化数据，需要事务保证 |
| 图关系（对象之间的关联网络） | Neo4j | 设备→备件→头寸→库房 的遍历查询，因果传导链推理 |
| 时序快照（月度历史数据） | PostgreSQL / TimescaleDB | 月度快照的时序查询（同比环比） |
| Agent 交互记录（对话和补全内容） | MongoDB | 非结构化对话记录，Agent补全的故障描述 |
| 向量索引（语义匹配用） | Milvus / Qdrant | 备件规格的语义匹配（Agent-3 替代件顾问查询） |

### 9.2 数据同步流

```
数据中台（通威ERP汇聚层）
  │ 每晚12点同步
  ▼
PostgreSQL（一级库头寸 + 备件主数据 + 采购单）
  │
  ├── → Neo4j（关系图谱：设备─使用─备件─跟踪─头寸─位于─库房）
  └── → Agent 工具层（供各Agent查询 + 写回）

扫码系统（二级库，实时录入）
  │ 实时写入
  ▼
PostgreSQL（二级库头寸 + 出入库记录）
  │
  └── → Agent-1 库存管家（接收出库事件 → 触发异步补全）
```
