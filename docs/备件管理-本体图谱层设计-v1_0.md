## 第一章 图谱在架构中的位置

### 1.1 双存储架构

```
┌─────────────────────────────────────────────────┐
│              本体定义（TBox，术语层）              │
│      类、属性、关系、规则、动作的语义定义          │
│           存储：本体编辑器 / 配置文件              │
└─────────────────┬───────────────────────────────┘
                  │ 指导实例数据的结构
                  ▼
┌─────────────────────────────────────────────────┐
│              实例数据（ABox，事实层）              │
│                                                  │
│   ┌─────────────────┐    ┌─────────────────┐    │
│   │   PostgreSQL     │    │     Neo4j       │    │
│   │  （精确事实库）   │◄──►│  （关系图谱）    │    │
│   │                  │同步│                  │    │
│   │  职责：           │    │  职责：          │    │
│   │  · 精确数量和金额 │    │  · 多跳关系遍历  │    │
│   │  · 出入库流水明细 │    │  · 影响范围扩散  │    │
│   │  · 采购单全字段   │    │  · 路径发现      │    │
│   │  · 月度快照时序   │    │  · 子图聚合      │    │
│   │  · 事务一致性     │    │  · 假设推演      │    │
│   └─────────────────┘    └─────────────────┘    │
│                                                  │
└─────────────────────────────────────────────────┘
```

**判断标准：什么查询走图谱、什么查询走 PostgreSQL？**

| 查询类型 | 走哪里 | 举例 |
|---|---|---|
| 沿关系遍历（2跳以上） | **图谱** | "3号线电机用哪些备件？这些备件的库存头寸在哪些库房？" |
| 影响范围扩散 | **图谱** | "如果6205轴承断货，影响哪些设备？" |
| 路径发现 | **图谱** | "这个呆滞件能被谁消化？经过什么替代关系？" |
| 精确数值查询 | **PostgreSQL** | "6205轴承在一级库的精确库存数量和金额" |
| 时序分析 | **PostgreSQL** | "6205轴承过去12个月的月度消耗趋势" |
| 事务性操作 | **PostgreSQL** | "出库扣减库存数量"（需要事务保证） |
| 全文搜索 | **PostgreSQL** | "搜索故障描述包含'振动'的出入库记录" |

**关键原则：PostgreSQL 是写入的权威源，Neo4j 是查询的加速层。所有数据先写 PostgreSQL，再同步到 Neo4j。**

## 第二章 图谱节点定义

### 2.1 节点类型总览

| 节点类型 | 中文 | 入图阶段 | 预估实例规模 | 说明 |
|---|---|---|---|---|
| InventoryPosition | 库存头寸 | α | ~600个（150种备件×4个库房） | 第一公民，图谱核心 |
| SparePart | 备件 | α | ~150个 | 物料主数据 |
| Warehouse | 库房 | α | 4个（1个一级+3个二级） | 固定节点 |
| Equipment | 设备 | α | ~50-100台 | 生产设备 |
| PurchaseOrder | 采购单 | α | ~20-50个（活跃状态的） | 已完成的归档 |
| StockMovement | 出入库记录 | α | ~3000个（近90天活跃） | 90天外归档到PostgreSQL |

**不入图的类：**
- 库存快照（InventorySnapshot）：纯时序数据，PostgreSQL 时序查询更高效
- 决策日志（DecisionLog）：审计记录，不需要图遍历

### 2.2 各节点冗余属性

只把**遍历过程中需要做过滤和判断的属性**同步到图谱节点上。其余属性留在 PostgreSQL，需要时按 ID 回查。

#### 库存头寸（InventoryPosition）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 头寸ID | string | 主键 |
| currentQty | 当前数量 | int | 遍历时需要判断"够不够" |
| safetyStock | 安全库存 | int | 遍历时需要判断"安不安全" |
| safetyGap | 安全缺口 | int | 遍历时需要过滤"哪些有风险" |
| availableQty | 可用数量 | int | 遍历时需要判断"实际能用多少" |
| inventoryValue | 库存金额 | decimal | 聚合计算总金额 |
| isStale | 是否呆滞 | boolean | 过滤呆滞头寸 |
| lastConsumedDate | 末次消耗日期 | date | 呆滞判断 |
| dataSource | 数据来源 | string | 区分一级库/二级库数据时效性 |

**不冗余的属性（留在 PostgreSQL）：** 库龄、月均消耗、消耗波动系数、周转率、可支撑天数、仓位位置、数据截止时间。需要时按头寸ID回查。

#### 备件（SparePart）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 物料编码 | string | 主键 |
| name | 备件名称 | string | 展示 |
| category | 物理分类 | string | 分类过滤 |
| criticality | 关键性等级 | string | 遍历时判断"这个件有多重要" |
| typicalLeadTime | 典型采购周期 | int(天) | 计算补货紧急度 |
| unitPrice | 标准价格 | decimal | 金额计算 |

#### 设备（Equipment）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 设备编号 | string | 主键 |
| name | 设备名称 | string | 展示 |
| location | 所在产线/工段 | string | 按产线过滤 |
| status | 运行状态 | string | 判断设备是否在用 |

#### 库房（Warehouse）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 库房ID | string | 主键 |
| name | 库房名称 | string | 展示 |
| level | 库房层级 | string | 区分一级/二级 |
| totalInventoryValue | 库存总金额 | decimal | 看板聚合 |
| targetValue | 目标金额 | decimal | 对比计算 |

#### 采购单（PurchaseOrder）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 采购单号 | string | 主键 |
| status | 状态 | string | 过滤"在途"采购单 |
| quantity | 采购数量 | int | 计算在途量 |
| expectedArrivalDate | 预计到货 | date | 判断"还要等多久" |

**不冗余的属性：** 采购单价、下单日期、实际到货日期、来源。

#### 出入库记录（StockMovement）节点

| 属性 | 中文 | 类型 | 冗余原因 |
|---|---|---|---|
| id | 记录ID | string | 主键 |
| movementType | 变动类型 | string | 过滤出库/入库/退库 |
| movementReason | 变动原因 | string | 按原因分类统计 |
| quantity | 数量 | int | 消耗聚合 |
| movementDate | 变动日期 | datetime | 时间范围过滤 |
| relatedEquipmentId | 关联设备ID | string | 关联到设备节点 |

**不冗余的属性：** 操作人、故障描述全文、Agent验证状态、关联采购单号。

## 第三章 图谱边定义

### 3.1 边类型总览

| 边类型 | 中文 | 起点 | 终点 | 方向 | 阶段 | 说明 |
|---|---|---|---|---|---|---|
| TRACKS | 跟踪 | 库存头寸 | 备件 | → | α | 头寸跟踪哪种备件 |
| LOCATED_IN | 位于 | 库存头寸 | 库房 | → | α | 头寸在哪个库房 |
| USES | 使用 | 设备 | 备件 | → | α | 设备使用哪些备件（BOM） |
| SUBSTITUTED_BY | 可替代 | 备件 | 备件 | → | β | 备件A可被备件B替代 |
| PROCURES | 采购 | 采购单 | 备件 | → | α | 采购单采购哪种备件 |
| AFFECTS | 影响 | 出入库记录 | 库存头寸 | → | α | 出入库记录影响哪个头寸 |
| CONSUMED_BY | 消耗于 | 出入库记录 | 设备 | → | α | 领用给了哪台设备 |

### 3.2 边上的属性

大部分边是纯结构关系，不需要属性。以下两种边需要属性：

**SUBSTITUTED_BY（可替代）边属性：**

| 属性 | 中文 | 类型 | 说明 |
|---|---|---|---|
| compatibility | 兼容度 | string | full（完全兼容）/ partial（部分兼容）/ emergency（应急可用） |
| condition | 适用条件 | string | 如"多粉尘环境更优""低负载场景下可用" |
| priceDiff | 价差 | decimal | 正值=替代件更贵，负值=更便宜 |

**USES（使用）边属性：**

| 属性 | 中文 | 类型 | 说明 |
|---|---|---|---|
| typicalQty | 典型用量 | int | 一次更换通常需要几个 |
| position | 安装位置 | string | 如"主轴承位""辅助轴承位" |

## 第四章 图谱分层结构

### 4.1 三层一图

整个图谱是一个 Neo4j 数据库，但逻辑上分三层，用**节点标签**区分：

```
┌─────────────────────────────────────────────────────────┐
│                     单一 Neo4j 图谱                      │
│                                                          │
│  ┌────────────────────────────────────────────────┐     │
│  │  结构层（Structure）                              │     │
│  │  标签：:Structure                                │     │
│  │                                                  │     │
│  │  设备 ──使用──→ 备件 ──可替代──→ 备件             │     │
│  │                  ↑                                │     │
│  │            库存头寸 ──跟踪──→                      │     │
│  │              │                                    │     │
│  │              └──位于──→ 库房                       │     │
│  │                                                  │     │
│  │  更新频率：月度（新增设备/备件/BOM/替代关系）       │     │
│  │  节点数量：~800（150备件+100设备+600头寸+4库房）   │     │
│  └────────────────────────────────────────────────┘     │
│                                                          │
│  ┌────────────────────────────────────────────────┐     │
│  │  状态层（Status）                                │     │
│  │  不是独立节点，是结构层节点上的属性                │     │
│  │                                                  │     │
│  │  头寸节点上：当前数量、安全缺口、是否呆滞、金额   │     │
│  │  库房节点上：总金额                               │     │
│  │  采购单节点上：状态、预计到货                      │     │
│  │                                                  │     │
│  │  更新频率：日度（一级库）/ 实时（二级库）          │     │
│  └────────────────────────────────────────────────┘     │
│                                                          │
│  ┌────────────────────────────────────────────────┐     │
│  │  事件层（Event）                                 │     │
│  │  标签：:Event                                    │     │
│  │                                                  │     │
│  │  出入库记录节点 ──影响──→ 库存头寸                │     │
│  │       │                                          │     │
│  │       └──消耗于──→ 设备                           │     │
│  │                                                  │     │
│  │  采购单节点 ──采购──→ 备件                        │     │
│  │                                                  │     │
│  │  更新频率：实时（新增出入库记录、采购单状态变化）  │     │
│  │  节点数量：~3000（近90天活跃）                    │     │
│  │  归档策略：90天以上的事件节点移到 PostgreSQL       │     │
│  └────────────────────────────────────────────────┘     │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 4.2 归档策略

| 归档对象 | 触发时机 | 操作 | 保留条件 |
|---|---|---|---|
| 出入库记录节点 | 每周日凌晨 | 变动日期 > 90天的节点及其边从 Neo4j 删除 | PostgreSQL 中完整保留 |
| 已完成采购单节点 | 每周日凌晨 | 状态=已到货 且 到货日期 > 90天的节点删除 | PostgreSQL 中完整保留 |
| 已取消采购单节点 | 每周日凌晨 | 状态=已取消 且 取消日期 > 30天的节点删除 | PostgreSQL 中完整保留 |

**归档不影响结构层和状态层。** 设备、备件、头寸、库房节点永远留在图谱中。

归档后，如果 Agent 需要查询 90 天以前的出入库记录（比如 Agent-7 做 12 个月消耗趋势），从 PostgreSQL 查。图谱只保证近期数据的遍历能力。

## 第五章 五种核心推演模式

### 5.1 模式1：正向影响扩散

**问题：如果 X 发生变化，会影响什么？**

**典型查询：设备健康度下降，影响哪些备件的哪些头寸？**

```
伪 Cypher 查询：

MATCH (e:Equipment {id: "LINE3_MOTOR"})
      -[:USES]->(sp:SparePart)
      <-[:TRACKS]-(pos:InventoryPosition)
RETURN sp.name AS 备件名称,
       pos.id AS 头寸ID,
       pos.currentQty AS 当前数量,
       pos.safetyGap AS 安全缺口,
       pos.inventoryValue AS 金额
```

**遍历路径：** 设备 →(使用)→ 备件 ←(跟踪)← 头寸

**结果示例：**

```
3号线电机的影响范围：
├── 6205轴承
│   ├── 一级总库头寸：28个，缺口=-26（充裕）
│   ├── 二级库A头寸：2个，缺口=0（临界）⚠️
│   └── 二级库B头寸：0个，缺口=2（不足）🔴
├── 温度传感器
│   ├── 一级总库头寸：15个，缺口=-13（充裕）
│   └── 二级库A头寸：8个，缺口=-6（充裕）
└── 密封圈
    └── 一级总库头寸：30个，缺口=-28（充裕）

结论：3号线电机故障时，6205轴承是最大风险点——二级库A临界、二级库B缺货。
```

**使用此模式的 Agent：**
- Agent-1 库存管家：领用时快速评估"领走这个件后，这台设备的保障状况如何"
- Agent-8 设备健康顾问（完整本体阶段）：设备健康度下降时计算连锁影响
- Agent-9b 场景推演师：模拟"如果这台设备故障率翻倍"的影响范围

### 5.2 模式2：反向溯源

**问题：这个结果的原因是什么？**

**典型查询：某个头寸安全缺口>0，为什么？是哪台设备消耗的？**

```
伪 Cypher 查询：

// 第一步：找到有风险的头寸
MATCH (pos:InventoryPosition)
WHERE pos.safetyGap > 0
WITH pos

// 第二步：找到最近30天消耗这个头寸的出入库记录
MATCH (sm:StockMovement)-[:AFFECTS]->(pos)
WHERE sm.movementType = 'OUT'
  AND sm.movementDate > date() - duration({days: 30})
WITH pos, sm

// 第三步：找到消耗关联的设备
MATCH (sm)-[:CONSUMED_BY]->(e:Equipment)
RETURN pos.id AS 头寸,
       e.name AS 消耗设备,
       count(sm) AS 近30天领用次数,
       sum(sm.quantity) AS 近30天领用总量
ORDER BY 近30天领用次数 DESC
```

**遍历路径：** 头寸 ←(影响)← 出入库记录 →(消耗于)→ 设备

**结果示例：**

```
二级库A的6205轴承头寸（安全缺口=0，临界）
溯源：
├── 3号线电机：近30天领用3次，共3个 ⚠️ 频次异常
├── 1号线电机：近30天领用0次
└── 2号线电机：近30天领用1次（正常）

根因分析：缺口主要由3号线电机的异常频次消耗导致。
建议：排查3号线电机根因，而不是增加采购量。
```

**使用此模式的 Agent：**
- Agent-1 库存管家：频次异常检测的底层查询
- Agent-5 采购建议师：生成采购建议时附带"为什么需要买"的溯源分析
- Agent-9a 日频优化：解释"为什么建议降/不降这个备件的库存"

### 5.3 模式3：路径发现

**问题：A 和 B 之间有什么关联？能不能走通？**

**典型查询：这个呆滞件能被谁消化？经过什么路径？**

```
伪 Cypher 查询：

// 找到呆滞件
MATCH (stalePart:SparePart)<-[:TRACKS]-(stalePos:InventoryPosition)
WHERE stalePos.isStale = true

// 找到它能替代的在用件
MATCH (stalePart)-[:SUBSTITUTED_BY]->(activePart:SparePart)

// 找到在用件被哪些设备使用
MATCH (e:Equipment)-[:USES]->(activePart)

// 找到这些设备最近的消耗频率
OPTIONAL MATCH (sm:StockMovement)-[:CONSUMED_BY]->(e)
WHERE sm.movementType = 'OUT'
  AND sm.movementDate > date() - duration({days: 90})
  AND sm.relatedEquipmentId = e.id

RETURN stalePart.name AS 呆滞件,
       stalePos.inventoryValue AS 呆滞金额,
       activePart.name AS 可替代的在用件,
       e.name AS 消耗设备,
       count(sm) AS 近90天消耗次数
ORDER BY 近90天消耗次数 DESC
```

**遍历路径：** 呆滞头寸 →(跟踪)→ 呆滞备件 →(可替代)→ 在用备件 ←(使用)← 设备 ←(消耗于)← 近期出入库记录

**结果示例：**

```
呆滞件：6205-2RS 密封型轴承（二级库A，180元，14个月未动）
消化路径：
├── 可替代 → 6205标准型轴承
│   ├── 3号线电机：近90天消耗9次（高频！最佳消化目标）
│   ├── 1号线电机：近90天消耗2次
│   └── 2号线电机：近90天消耗3次
│
└── 建议：下次3号线电机领用6205轴承时，Agent-1推荐用这个呆滞件替代。
    预计1-2周内可消化。
```

**使用此模式的 Agent：**
- Agent-4 呆滞侦探：为每个呆滞件找消化路径
- Agent-3 替代件顾问：确认替代关系链（可能多跳：A替代B、B替代C）
- Agent-1 库存管家：领用时检查"有没有呆滞件可以用"

### 5.4 模式4：子图聚合

**问题：满足某个条件的子图的统计值是什么？**

**典型查询：某个库房下所有头寸的金额汇总、呆滞占比、风险头寸数**

```
伪 Cypher 查询：

MATCH (pos:InventoryPosition)-[:LOCATED_IN]->(wh:Warehouse {id: "WH_PRIMARY"})
RETURN wh.name AS 库房,
       count(pos) AS 头寸总数,
       sum(pos.inventoryValue) AS 总金额,
       sum(CASE WHEN pos.isStale THEN pos.inventoryValue ELSE 0 END) AS 呆滞金额,
       sum(CASE WHEN pos.safetyGap > 0 THEN 1 ELSE 0 END) AS 风险头寸数,
       sum(CASE WHEN pos.safetyGap > 0 THEN pos.inventoryValue ELSE 0 END) AS 风险金额
```

**典型查询：本月各产线的消耗金额排名**

```
伪 Cypher 查询：

MATCH (sm:StockMovement)-[:CONSUMED_BY]->(e:Equipment)
WHERE sm.movementType = 'OUT'
  AND sm.movementDate >= date("2026-03-01")
WITH e.location AS 产线, sum(sm.quantity) AS 消耗数量
MATCH (sm2:StockMovement)-[:CONSUMED_BY]->(e2:Equipment {location: 产线})
      -[:AFFECTS]->(pos:InventoryPosition)-[:TRACKS]->(sp:SparePart)
RETURN 产线,
       sum(sm2.quantity * sp.unitPrice) AS 消耗金额
ORDER BY 消耗金额 DESC
```

**使用此模式的 Agent：**
- Agent-2 金额看板员：分库房/分工段/分工序的金额聚合
- Agent-9a 日频优化：全局扫描所有头寸计算可优化空间
- Agent-4 呆滞侦探：呆滞件的分类汇总（按原因、按库龄段）

### 5.5 模式5：假设推演（影子图）

**问题：如果改变某个参数，整体会怎么变？**

**核心设计：不在 Neo4j 原图上修改，而是提取子图到内存中做假设计算。**

```
推演流程：

第一步：从 Neo4j 提取相关子图
  MATCH (pos:InventoryPosition)-[:TRACKS]->(sp:SparePart)
        -[:USES]-(e:Equipment)
  WHERE sp.id = "BJ-0421"  // 6205轴承
  RETURN pos, sp, e
  → 拿到所有与6205轴承相关的头寸、设备

第二步：在内存中构建影子图（轻量副本）
  复制所有相关节点和边的属性值
  这是一个临时的数据结构，不持久化

第三步：在影子图上修改参数
  假设：safetyStock 从 2 改为 1
  重算：每个头寸的 safetyGap = 1 - availableQty
       每个头寸的"释放量" = currentQty - 1
       释放金额 = 释放量 × unitPrice

第四步：评估影响
  遍历关联设备：如果安全库存降到1，各设备的保障天数变化
  计算风险：基于历史消耗速度，缺货概率从 X% 变为 Y%

第五步：返回推演结果（不写回原图）
  输出：
  {
    "假设": "6205轴承安全库存从2降为1",
    "释放金额": 800,        // 4个库房各释放1个 × 200元
    "缺货风险变化": "3%→12%",
    "影响设备": ["1号线电机", "2号线电机", "3号线电机"],
    "高风险设备": "3号线电机（近期消耗频次异常，降库存风险高）",
    "建议": "1号线和2号线可安全降至1个，3号线建议维持2个"
  }
```

**使用此模式的 Agent：**
- Agent-9a 日频优化：每日计算"哪些备件可以安全降库存"
- Agent-9b 场景推演师（完整本体阶段）：复杂的多参数假设推演

**关键原则：影子图计算的结果只是建议，不自动写回。必须经过人工确认（工段长审批）后，才通过动作 A 系列写回真实数据。**

## 第七章 图谱与 PostgreSQL 同步机制

### 7.1 同步方向与策略

```
┌───────────────────────────────────────────────────────┐
│                      同步全景                          │
│                                                       │
│  数据中台 ──每晚0:00──→ PostgreSQL ──批量──→ Neo4j     │
│  （通威ERP）           （一级库数据）   （一级库节点属性更新）│
│                                                       │
│  扫码系统 ──实时──→ PostgreSQL ──实时──→ Neo4j         │
│  （二级库）          （出入库记录）    （新增事件节点+      │
│                                      头寸属性更新）     │
│                                                       │
│  本体编辑器 ──发布时──→ Neo4j                          │
│  （结构变更）           （新增/删除节点类型、边类型）     │
│                                                       │
│  Neo4j ──不回写──→ PostgreSQL                          │
│  （图算法结果是临时的，留在应用层）                      │
└───────────────────────────────────────────────────────┘
```

### 7.2 分层同步策略

| 层 | 同步触发 | 同步内容 | 延迟容忍 |
|---|---|---|---|
| **结构层** | 本体编辑器发布新版本时 | 新增/删除节点类型、边类型、BOM关系变更 | 分钟级 |
| **状态层（一级库）** | 每晚数据中台同步完成后 | 一级库所有头寸节点的属性更新（数量、金额、安全缺口、是否呆滞） | T-1（一天延迟，客户接受） |
| **状态层（二级库）** | 出入库动作执行后 | 二级库相关头寸节点的属性更新 | 秒级（准实时） |
| **事件层（一级库）** | 每晚数据中台同步完成后 | 新增出入库记录节点和边 | T-1 |
| **事件层（二级库）** | 出入库动作执行后 | 新增出入库记录节点和边 | 秒级（准实时） |
| **事件层归档** | 每周日凌晨 | 删除90天以上的事件节点 | 周级 |

### 7.3 同步流程详解

#### 每晚批量同步（一级库数据）

```
触发：数据中台同步到 PostgreSQL 完成后（约每晚 0:30）

步骤：
1. 查询 PostgreSQL 中所有一级库头寸的最新状态
2. 批量更新 Neo4j 中对应头寸节点的属性
   （当前数量、安全缺口、库存金额、是否呆滞、末次消耗日期）
3. 查询 PostgreSQL 中当天新增的一级库出入库记录
4. 在 Neo4j 中创建对应的出入库记录节点和边
   （AFFECTS→头寸，CONSUMED_BY→设备）
5. 重算所有库房节点的 totalInventoryValue
6. 记录同步日志（同步时间、处理条数、成功/失败）

预估耗时：< 5分钟（~600个头寸节点 + ~50条新出入库记录）
```

#### 实时同步（二级库扫码）

```
触发：二级库出入库动作（A01/A02）在 PostgreSQL 中执行完成后

步骤：
1. 在 Neo4j 中创建出入库记录节点
2. 创建 AFFECTS 边连接到对应头寸
3. 如果有关联设备，创建 CONSUMED_BY 边连接到设备
4. 更新头寸节点的属性（当前数量、安全缺口、末次消耗日期）
5. 更新库房节点的 totalInventoryValue

预估耗时：< 500ms（单条记录）
```

#### 结构变更同步

```
触发：本体编辑器发布新版本时

典型场景：
- 新增一台设备 → Neo4j 中添加设备节点 + USES 边
- 新增一种备件 → 添加备件节点 + 为每个库房创建头寸节点 + TRACKS/LOCATED_IN 边
- 新增替代关系 → 添加 SUBSTITUTED_BY 边（含兼容度等属性）
- 备件关键性等级更新 → 更新备件节点的 criticality 属性

频率：很低，一个月可能几次
```

### 7.4 一致性保证

| 场景 | 策略 |
|---|---|
| PostgreSQL 写入成功但 Neo4j 同步失败 | 重试3次；仍失败则记入异常队列，不阻塞业务 |
| Neo4j 和 PostgreSQL 数据不一致 | 以 PostgreSQL 为准。每日批量同步时会覆盖 Neo4j 的状态层属性 |
| 二级库实时同步期间 Neo4j 不可用 | 出入库操作照常在 PostgreSQL 中完成（不阻塞业务），Neo4j 恢复后补同步 |
| 归档操作删除了 Agent 正在查询的节点 | 归档在凌晨低峰期执行，与 Agent 运行时间错开 |

**核心原则：图谱不可用不阻塞业务操作。PostgreSQL 是权威源，图谱是查询加速层。最坏情况下 Agent 降级为直接查询 PostgreSQL（速度慢但结果正确）。**

## 第八章 演化路径

### 8.1 MVO-α 图谱（Day-1）

```
节点：库存头寸(~600) + 备件(~150) + 库房(4) + 设备(~50) + 出入库记录(近90天)
边：跟踪 + 位于 + 使用 + 影响 + 消耗于 + 采购
推演模式：正向扩散 + 反向溯源 + 子图聚合
不支持：替代关系（substitutedBy 边还没有数据）、假设推演（Agent-9a先用PostgreSQL直算）
```

### 8.2 MVO-β 图谱（3-6个月后）

```
新增边：SUBSTITUTED_BY（替代关系，含兼容度属性）
新增能力：路径发现（呆滞件消化路径）、假设推演（影子图）
新增索引：备件节点的 criticality 属性索引（支持按关键性过滤）
```

### 8.3 完整本体图谱（12个月后）

```
新增节点：故障事件（FaultEvent）
新增边：发生于（occursOn）、触发（triggers）
新增推演：因果传导链（设备健康→故障概率→备件消耗→头寸安全水位→金额影响）
数据来源：Agent-1 在12个月内积累的故障描述+设备关联数据 迁移为正式的FaultEvent节点
```

### 8.4 图谱规模预估

| 阶段 | 节点数（预估） | 边数（预估） | 说明 |
|---|---|---|---|
| MVO-α | ~4000 | ~8000 | 800结构节点 + ~3000事件节点（近90天） |
| MVO-β | ~4200 | ~8500 | +替代关系边 |
| 完整本体 | ~5000 | ~10000 | +故障事件节点 |

这个规模对 Neo4j 来说非常小——查询延迟在个位数毫秒级，不需要特别的性能优化。

| 查询场景 | 延迟要求 | 原因 |
