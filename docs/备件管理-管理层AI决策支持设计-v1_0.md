## 第一章 问题诊断：管理者需要什么

### 1.1 管理者每天面对的四个问题

永祥科技的备件管理者（工段长、管理层）有一个明确的考核目标：**总库存金额从800万降至560万，降幅30%。** 围绕这个目标，他们每天都在回答四个问题：

**问题一："812万的库存里，到底哪些钱该花、哪些不该花？"**

现状：管理者只能看到一个总金额。812万是多了还是少了？他不知道。因为他无法区分——里面有多少是正在流转支撑生产的"活钱"、多少是合理的安全储备、多少是过度采购的浪费、多少是设备淘汰后的沉淀。一刀切砍30%会出事，但不砍完不了KPI。

**问题二："252万的缺口，从哪里挤、按什么顺序挤、多久能挤完？"**

现状：管理者没有路线图。每个月开会时看看金额降了多少，降得不够就施压——但压谁、压什么、压多少，没有依据。Agent-9a给了一张"今日可降4.8万"的清单，但管理者需要的不是日度战术动作，而是一条跨越6个月的达标路径。

**问题三："我上个月做的决策对不对？哪些见效了、哪些没见效？"**

现状：决策是一次性的——采购批了就批了，降库存降了就降了。没有人帮管理者回头检验决策效果。上月批准了8条采购建议，到了没有？用了没有？买多了还是买少了？这些信息散落在ERP和Excel里，没有人整合。

**问题四："下属说这个不能降，他说得对不对？"**

现状：管理者和一线工程师之间存在信息不对称。工程师说"PLC模块绝对不能动"，管理者无法独立验证——他不了解每个备件的关键性、替代性、消耗规律和采购周期。他要么全信（KPI完不成），要么全不信（出安全问题）。

### 1.2 为什么现有设计没有解决这些问题

| 现有设计 | 回答了什么 | 没回答什么 |
|---|---|---|
| Agent-2 金额看板 | "总金额是多少、环比涨跌" | 金额的结构是什么、该砍哪里 |
| Agent-9a 日频优化 | "今天可以降4.8万" | 6个月后能不能达标、路径上有什么风险 |
| Agent-4 呆滞侦探 | "这些件呆滞了" | 呆滞是症状还是根因？是过度采购还是设备淘汰？ |
| Agent-5 采购建议 | "该买这些" | 买了之后整体金额趋势怎么变 |

**根本问题：这些Agent各自回答局部问题，没有一个Agent在全局层面帮管理者做战略思考。**

## 第二章 Agent-10：库存策略顾问

### 2.1 定位

Agent-10 不是Agent-2（看板）和Agent-9a（日频优化）的合体。它是一个全新的**策略层角色**。

| Agent | 比喻 | 服务对象 | 回答什么 |
|---|---|---|---|
| Agent-2 金额看板 | 温度计 | 所有人 | 现在多少度 |
| Agent-9a 日频优化 | 日程表 | 工段长 | 今天该做什么 |
| **Agent-10 策略顾问** | **军师** | **管理层+工段长** | **仗该怎么打、打到哪一步了、上一仗打得如何** |

### 2.2 IDENTITY

### 2.3 四大核心能力

#### 能力一：库存结构透视

**输入：** 全部库存头寸的当前状态 + 历史消耗数据 + 备件关键性 + 采购周期

**计算逻辑：**

将每一个库存头寸分类到四个口袋中：

```
口袋一：活库存（正在流转、支撑生产的钱）
  条件：近3个月有消耗 且 当前库存 ≤ 建议安全库存 × 2
  含义：这些钱必须花，砍了会缺货

口袋二：安全储备（为防万一备的钱）
  条件：关键性=A 且 (采购周期 > 30天 或 设备停机损失极高)
  含义：这些钱是保险费，不能轻易砍

口袋三：可优化过剩（多备了的钱，可以安全释放）
  条件：当前库存 > 建议安全库存 × 2 且 不属于口袋二
  含义：优化主战场——降库存主要从这里挤

口袋四：呆滞沉淀（躺着不动的钱）
  条件：isStale = true（库龄>12个月无消耗）
  含义：需要处置——代用、调拨、折价、报废
```

**输出示例：**

```
库存结构透视（截至2026年3月19日）

总库存金额：812万元

┌──────────────────────────────────────────────────┐
│                                                    │
│  ████████████████ 活库存      312万  38.4%         │
│  ██████████       安全储备    186万  22.9%         │
│  ████████████████ 可优化过剩  228万  28.1%  ← 主战场│
│  █████            呆滞沉淀     86万  10.6%         │
│                                                    │
│  目标：560万                                        │
│  需释放：252万                                      │
│  可释放空间：228万（过剩）+ 86万（呆滞）= 314万     │
│  空间充裕度：314万 > 252万 ✅                        │
│                                                    │
│  结论：目标可达。不需要动活库存和安全储备。           │
│  从过剩和呆滞中释放252万即可。                       │
└──────────────────────────────────────────────────┘
```

**LLM 增值：自然语言解读**

```

这是维持生产运转的底线，不应触碰。

真正需要关注的是另外314万：
· 228万的过剩库存：主要集中在轴承类（92万）和通用传感器类（68万），
  这些备件采购周期短（7-14天）、替代件多、消耗稳定，完全可以安全降库存。

  28万是过度采购的结果（建议代用匹配消化），26万需要逐件评估。
```

#### 能力四：判断验证

**交互模式：** 管理者主动提问，Agent-10 基于图谱数据做独立分析。

**场景示例1：验证"不能降"的判断**

```
管理者："工段长说PLC模块不能降，Y型阀门也不能降，你怎么看？"

✦ Agent-10：

"逐个评估：

1. 进口PLC模块（2个，16万元）——工段长说得对 ✅
   · 关联设备：3台产线PLC柜（关键性=A）
   · 故障后果：单线停机，每小时产能损失约X万元
   · 采购周期：90天（进口德国）
   · 近12月消耗：1个
   · 结论：16万是保险费，停机1天的损失就远超16万。维持。

2. Y型阀门（3个，4500元）——建议挑战 ⚠️
   · 关联设备：2号线冷却系统（关键性=B）
   · 故障后果：降速运行，不停机
   · 采购周期：7天（国内供应商）
   · 近12月消耗：0个
   · 替代件：有2个兼容型号在库（Z型阀门3个、W型阀门2个）
   · 结论：3个Y型阀门零消耗+7天就能买到+有5个替代件在库。
     建议降到1个，释放3,000元。风险极低。

总结：PLC模块16万不动，Y型阀门可释放3,000元。
工段长对关键件的判断是准确的，但对通用件偏保守。"
```

**场景示例2：验证"应该买"的判断**

```
管理者："采购说3号线变频器要加急买3台，每台8000元，有必要吗？"

✦ Agent-10：

"3号线变频器加急采购评估：

当前库存：5台（安全库存=2）
近3月消耗：3台（月均1台，为正常水平的2倍）
消耗溯源：
  · 1月：1号线变频器更换（正常老化）
  · 2月：3号线变频器更换（电路板故障）
  · 3月：3号线变频器更换（又是电路板故障）

⚠️ 异常发现：3号线连续2个月更换变频器，且故障模式相同（电路板故障）。
这不像是正常消耗，更像是3号线存在一个根因问题导致变频器加速损坏。

建议：
  · 不反对采购——当前库存5台，如果3号线问题持续，2-3个月就用完
  · 但更重要的是：排查3号线变频器电路板故障的根因
  · 如果根因是供电质量问题或散热不足，加装保护或改善散热后，
    变频器消耗会回到正常水平（月均0.5台），就不需要加急采购
  · 建议：先批准买2台（而不是3台）应急，同时安排3号线电气检查。
    如果根因解决了，剩余库存5+2=7台可支撑14个月。"
```

这个回答的价值不是"买还是不买"，而是**揭示了采购需求背后的根因问题**。管理者看到后会说"不只是批采购的事，要安排人去查3号线"。这才是管理级别的洞察。

### 2.4 Agent-10 的数据依赖

| 依赖数据 | 来源 | 何时可用 |
|---|---|---|
| 全部头寸的当前库存和金额 | PostgreSQL（数据中台同步） | Day-1 |
| 备件关键性等级 | 本体（共创后填充） | MVO-β（3-6月后）；Day-1用简化版（按单价+采购周期粗分） |
| 近3月消耗数据 | StockMovement记录 | 上线3个月后逐步可用 |
| 替代件关系 | 本体（substitutedBy） | MVO-β |
| 设备-备件关联 | 图谱（USES边） | Day-1（BOM数据导入） |
| 决策日志 | DecisionLog | MVO-β |
| 采购闭环数据 | PurchaseOrder | 上线后逐步积累 |

**关键：Agent-10 的"库存结构透视"能力 Day-1 就能用**——它只需要头寸数据+备件价格+采购周期，这些数据在ERP里都有。关键性等级在Day-1用简化规则替代（单价>5万=A类，单价1000-50000且采购周期>30天=B类，其余=C类）。

达标路径规划和决策回溯需要3个月以上的消耗数据才有意义。判断验证能力从Day-1就可以部分使用（基于BOM和采购周期做静态评估），3个月后加入消耗趋势分析变得更强。

### 2.5 Agent-10 的进化路径

| 时间 | 能力 | 依赖 |

## 第三章 管理层工作台重设计

现有设计是"速览卡片+待办清单"——这是操作型工作台。管理层需要的是**决策型工作台**。

### 3.1 设计理念

管理者打开系统，第一眼看到的三个区域：

```
┌──────────────────────────────────────────────────────────────┐
│  ① 我在哪：目标达成现状                                       │
│  ② 下一步：接下来该做什么                                     │
│  ③ 回头看：上次决策效果如何                                   │
└──────────────────────────────────────────────────────────────┘
```

不是"有什么待办推给你"（那是被动响应），而是"你的战略目标进展到哪了"（这是主动掌控）。
