## 二、核心痛点（按客户优先级）

### 痛点1：二级库房管理混乱（最痛）

> 二级库房为人工管理，没有规范化、系统化。

**具体表现：**
- 从一级库领用的备件经常堆积在二级库，不知道什么时间、什么人领用的
- 领用数量与使用量、库存量数量对不上
- 经常出现多拿偷拿配件的情况
- 退库不记录——"可能直接就放在二级备件库里面了"
- 二级库没有统计过金额

**来源：** 调研人员1/2核心痛点+调研人员3 Q9

### 痛点2：库存金额难核对（最急）

> 无法进行同比环比分析，无法直观查看总体与明细数据。

**具体表现：**
- 库存金额无法同比环比，难以体现与去年同期相比的改善
- 一级库领用金额与二级库实际使用金额不一致
- 螺丝、轴承等小配件领用后存放混乱，导致金额无法准确核对
- 有明确考核指标（总库存金额降低30%），但指标无法落地
- 统计领用金额、入库金额需要手工做报表，经常加班

**来源：** 调研人员1/2核心痛点+调研人员3 Q1/Q17

### 痛点3：考核指标悬空

| 指标 | 目标 | 现状 |
|---|---|---|
| 总库存金额 | 降低30%（如1000万→700万） | 无工具分解和追踪 |
| 设备故障率 | 降低20%~30% | 故障记录是纸质的，无法量化 |
| 完成压力 | 扣绩效 | "任务要尽量达成，不然扣绩效" |

**来源：** 调研人员1/3

### 4.2 二级库扫码出入库流程（已设计，待实现）

**二级库入库流程：**
```
扫码 → 选择关联出库单 → 备件入库（确认数量）→ 生成出入库记录（出库单ID、入库数量、出库记录） → 直接入库
```

**补充规则：** 系统每天凌晨12点同步出库单据，规定只看前一天的数据。用户从一级库拿走出库使用后，如果已使用则不入二级库（生成出库记录标记"已使用"），如果未使用则录入二级库。

**二级库出库流程：**
```
扫码 → 备件出库（确认数量）→ 生成出库记录 → 更新库存
```

**来源：** 补充材料流程图

### 4.3 采购流程

```
一级库低于安全库存 → 人工发现（Excel）→ 提申请 → 工段长审批
→ 定期采购（每月5号/20号）或 加急采购
→ 采购部门询价（对比三家）→ 下单
→ 供应商响应（2天 ~ 半年不等）→ 到货 → 验收入库
```

**关键时间点：**
- 采购申请节点：每月5号和20号为固定节点，其他时段属于加急采购
- 采购周期跨度极大：最短2天，最长半年（缺货或现生产）
- 最耗时环节：采购询价（对比三家）+ 供应商物流
- 采购部门与备件部门是分离的，备件部门只能提申请

**来源：** 调研人员2/3

## 六、已规划需求模块

客户已有明确的需求规划，5个功能模块 + 1个看板：

### 6.1 备件台账

- 备件主数据集中管理（目录、规格、分类、适用设备、库存属性、金额、安全库存、替代件、供应商）
- 是所有其他模块的数据底座
- 适用角色：备件管理员、仓库管理员、设备工程师、采购人员、系统管理员

### 6.2 备件库房管理

- 一级库 + 二级库一体化管理
- 覆盖：库存查询、出库、入库、退库、报废、盘点、交叉申领、在途与验收
- **重点支撑二级库业务（最大痛点）**
- 出入库记录需要增加**导出功能**：分析备件用在了哪些地方/金额多少、当月当周当日统计

### 6.3 备件预留

- 保养计划确认后自动预留备件（锁定可用量）
- 任务取消/完成/超时后自动释放
- 支持手工预留与审批
- **前置条件：保养计划需含备件需求清单——当前不具备**

### 6.4 备件采购需求清单

- 汇总自动生成 + 人工确认的采购需求
- 低库存预警 → 请购建议 → 主管确认/调整/审批 → 提交采购系统
- **前置条件：安全库存分析已完成 + ERP采购接口可用**

### 6.5 呆滞库存

- 识别、分析、处置呆滞备件
- 展示呆滞金额、占比、库龄分布、优化建议
- 支撑动作：调降安全库存、替代、调拨、报废、变卖

### 6.6 看板需求（调研人员2）

| 看板项 | 说明 |
|---|---|
| 组织维度 | 分部门、分工段、分工序展示 |

| H2 | 备件有清晰的A/B/C分类体系 | **⚠️ 部分推翻** | 有物料编码和物理分类，无管理分级 | criticality 属性需共创 |
| H3 | 库存管理到库位级别 | **✅ 降级保留** | 有仓位属性，但二级库形同虚设 | StorageSlot 移出 MVO |
| H4 | 消耗侧只有故障维修和保养 | **❌ 推翻** | 至少6类：故障/保养/技改/项目/调拨/报废 | StockMovement 类型扩展 |
| H5 | 出库记录=真实消耗 | **❌ 二级库推翻** | 一级库相对准确；二级库无记录、多拿偷拿 | 二级库建立记录流程是前置条件 |
| H6 | 保养计划包含备件清单 | **❌ 推翻** | 只写保养部位，不写备件 | MaintenancePlan 降级 |
| H7 | 工厂间存在调拨机制 | **❌ 推翻（有期望）** | 无机制，查不到其他厂库存，但业务线期望 | 调拨 Action 移出 MVO，结构预留 |
| H8 | 故障记录是电子化的 | **❌ 推翻** | 纸质 | FaultEvent 暂不纳入 MVO |
| H9 | 历史数据深度>12个月 | **✅ 验证** | 一周内可从ERP拿到 | 一级库行为画像可算 |
| H10 | 有明确的安全库存和呆滞标准 | **⚠️ 需共创** | 安全库存=统一2个；呆滞=1年，但定义未对齐 | 规则做成可配置参数 |

**总结：10个假设中，6个被推翻，2个需修正，2个被验证。**

## 九、数据红线清单

以下是确认的数据缺失，每项附带影响评估：

| # | 红线 | 影响 | 应对策略 |
|---|---|---|---|
| 🔴 1 | 故障记录为纸质，无电子化 | 故障概率模型(H1)不可行；故障-备件因果链无法建立 | Agent交互中逐步积累故障描述；推动客户电子化 |
| 🔴 2 | 二级库从未统计过库存数量和金额 | 一级库+二级库的总金额无法核对；二级库头寸是黑箱 | 扫码流程上线是前置条件 |
| 🔴 3 | 保养计划不含备件清单 | 保养需求退化为不确定性信号；预留功能前置条件不满足 | 第二阶段帮客户建保养方案模板 |
| 🟡 4 | 出库不记录故障原因 | 消耗分析无法区分"故障消耗"和"保养消耗" | 系统设计时要求选择领用原因 |
| 🟡 5 | 呆滞定义客户内部未对齐 | 呆滞标记规则可能有争议 | 需二次确认对齐 |
| 🟡 6 | 采购周期跨度极大（2天~半年） | 安全库存计算中采购周期不能当常量 | 按备件类别分别统计历史采购周期 |
| 🟡 7 | 跨工厂库存不可见 | 调拨机制不可行；补给侧只剩采购一条路 | 安全库存计算要更保守 |
| 🟢 8 | IoT数据存在但未对接 | 设备健康度、RUL预测暂不可行 | 列入未来接口规划 |

## 十、MVO 边界重划

基于以上裁决，MVO 分两阶段：

### MVO-α：数据底座（第一优先级）

| 类别 | 内容 |
|---|---|
| 核心类 | InventoryPosition（库存头寸）、SparePart（备件）、Warehouse（库房，含一级/二级）、StockMovement（出入库记录）、PurchaseOrder（采购单） |
| 新增设计 | StockMovement 增加 warehouseLevel（一级/二级）、movementReason 枚举（故障维修/保养/技改/项目/调拨/报废退换）、source（ERP同步/扫码录入） |
| 新增类 | InventorySnapshot（库存快照，月度记录数量和金额，支撑同比环比） |
| 关系 | tracks、locatedIn、affects、procures |
| 规则 | R01 安全库存预警（阈值=2）、R03 呆滞标记（库龄>12月） |
| 数据对接 | 数据中台每日同步（T-1） + 二级库扫码实时 |

### MVO-β：规则引擎（α稳定后叠加）

| 类别 | 内容 |
|---|---|
| 新增类 | DecisionLog（决策日志） |
| 新增关系 | SparePart → substitutedBy → SparePart（替代件） |
| 规则扩展 | 差异化安全库存（A/B/C共创）、呆滞处置建议、低库存请购建议 |
| Action | 生成请购建议（人工确认）、标记呆滞（自动）、触发预警（自动） |

### 推迟到完整本体阶段

| 内容 | 解锁条件 |
|---|---|
| FaultEvent、FaultProbability | 故障记录电子化 |
| MaintenancePlan → requires → SparePart | 保养方案模板建立 |
| 跨工厂调拨 | 多工厂库存可见性打通 |
| AI需求预测、设备生命周期 | 12个月二级库数据 + 故障数据电子化 |
| 复杂What-If推演 | 完整本体 + 规则引擎稳定 |

## 十一、Agent 系统三层架构

### 第一层：Day-1 可上线（5个Agent）

| Agent | 核心职责 | 确定性部分（代码） | 不确定性部分（LLM） |
|---|---|---|---|
| **Agent-1 库存管家** | 基于出入库记录做智能补全+查询+分析（扫码负责写，Agent负责读+分析+补全） | 库存计算、出库单关联、金额计算 | 从出库上下文推断故障原因和关联设备；异步轻量确认 |
| **Agent-2 金额看板员** | 分部门/工段/工序金额视图，同比环比，领用Top3，金额vs目标对比 | 金额加总、环比同比计算、排名、快照生成 | 自然语言分析解读："本月X工段领用上升18%，主因是..." |
| **Agent-3 替代件顾问** | 缺货或呆滞时推荐可替代型号（代用匹配） | 规格参数匹配、兼容性规则 | NLP解析技术文档提取规格；模糊推理 |
| **Agent-4 呆滞侦探** | 扫描库存标记呆滞，生成处置建议 | 库龄计算、呆滞标记 | 分析呆滞原因，推荐处置路径，联动Agent-3 |
| **Agent-9a 日频优化建议** | 基于当日出库+当前库存，算可降数量/金额/建议清单 | 纯确定性计算 | 将计算结果翻译为管理者可理解的建议和风险提示 |

### 第二层：MVO-β 阶段（3-6个月后）

| Agent | 解锁条件 |
|---|---|
| **Agent-5 采购建议师** | A/B/C分类共创完成 + 二级库3个月数据 + 至少一轮采购闭环 + 总部ERP采购接口可用 |
| **Agent-6 预留协调员** | 保养方案模板建立 + Agent-1积累的保养-备件对应关系 |

### 第三层：完整本体阶段（12个月后）

| Agent | 解锁条件 |
|---|---|
| **Agent-7 需求预测师** | 12个月二级库数据 + 保养关联 + 故障关联 |
| **Agent-8 设备健康顾问** | 故障电子化 + IoT对接 + 6个月设备运行数据 |
| **Agent-9b 场景推演师** | Agent-5/7稳定 + 规则引擎校准 + 完整预测→采购→验证闭环 |

### 数据飞轮机制

```
Agent-1（库存管家）— 每次出入库交互
  ├─ 积累 "故障-备件-设备" 关联 ──→ 12个月后解锁 Agent-7/8
  ├─ 区分消耗场景类型 ──→ 3个月后解锁 Agent-5
  └─ 标记保养领用 ──→ 6个月后解锁 Agent-6

Agent-4（呆滞侦探）
  └─ 调用 Agent-3（替代件）──→ 消耗呆滞库存 ──→ 释放资金 ──→ 趋近30%目标

Agent-5（采购建议）
  └─ 采购闭环数据 ──→ 校准采购周期参数 ──→ 参数稳定后解锁 Agent-9b
```

**核心设计原则：** Agent-1是所有后续Agent的数据源头。它在Day-1上线后看似只是"帮你查库存、补全记录"的工具，但它在静默积累整个AI体系所需的训练数据。12个月后回头看，那些"顺便"采集的故障描述、设备关联、消耗场景分类，就是需求预测和设备健康模型的基础。
